# 分层记忆系统 - 面试快速回忆卡（口语版）

> **使用说明**：面试前10分钟快速复习，面试中回答问题前快速回忆
> 每个答案约30-40秒说完，语言更口语化，适合面试表达
> **加粗关键词**便于快速扫描和记忆

---

## 🔥 TOP 10 高频必考

### Q1: 为什么 Agent 需要"记忆"？不用不行吗？

**30秒答案**:
其实一开始我也试过不记忆，每次都让用户重新输入所有信息，比如幻灯片数量、语言偏好、模板类型。但体验很差，用户每次生成都要重复说同样的话。

后来我加了记忆系统，记住用户的基本偏好。这样用户只需要说"做一个关于 AI 的 PPT"，系统会自动用他之前设置过的15页、中文、专业语调。从用户反馈看，**留存率提升了 30%**，因为个性化体验好。

记忆的核心价值是**减少重复输入**和**提升连续性**。没有记忆，每次都是新对话，体验像失忆一样。

---

### Q2: 为什么不直接把历史对话全塞进 prompt？

**30秒答案**:
我一开始也试过全量历史，但很快就发现问题。首先是 **token 爆炸**，长对话之后 prompt 能到 5000+ tokens，成本很高。其次是 **latency 很高**，每次请求都要处理大量历史文本，响应时间超过 10 秒。还有就是 **上下文污染**，早期的无关信息会干扰当前的生成。

所以我做了个折中：只总结关键信息存到用户偏好里，比如语言、页数、语调这些长期有效的配置。短期的上下文用 LangGraph State 在 Agent 间传递。这样既能记住用户习惯，又不会让 prompt 膨胀。

从数据看，token 使用量下降了 **70%**，响应速度提升 **3 倍**。

---

### Q3: 为什么选"分层存储"？直接一个数据库不行吗？

**30秒答案**:
我一开始确实只用 MySQL，但发现高频读取场景下性能不够。用户偏好这种数据，每次生成 PPT 都要查，QPS 能到 100+。MySQL 的延迟大概 **5-15 毫秒**，而且每次都要建立连接，并发高的时候会成为瓶颈。

后来我加了 Redis 缓存层，延迟降到 **0.5-1 毫秒**，而且是纯内存操作，QPS 能轻松上 10000。我用的是 **Cache Aside 模式**，读的时候先查 Redis，miss 了再查 MySQL，然后回填缓存。写的时候先更新 MySQL，再删除缓存。

这个组合兼顾了性能和可靠性：MySQL 提供持久化和事务支持，Redis 提供低延迟和高并发。从压测数据看，P50 延迟从 12ms 降到 0.8ms，P99 从 50ms 降到 3ms。

---

### Q4: 为什么用 Redis + MySQL 这个组合？

**30秒答案**:
选这个组合是因为它们**互补**。Redis 是纯内存的，延迟低、QPS 高，但没有持久化，重启就丢数据。MySQL 延迟高一些，但数据可靠，支持事务。

我的策略是：**热数据放 Redis，冷数据 MySQL 持久化**。用户偏好这种高频读取的数据，Redis 扛主流量，MySQL 做兜底。Redis 挂了就降级到 MySQL，性能会变慢但不会宕机。

从技术选型看，Redis 的单线程模型避免了并发竞争，MySQL 的 InnoDB 引擎支持 ACID，这个组合在生产环境很成熟。而且运维成本可控，两个都是开源社区的主流方案。

---

### Q5: 你的缓存策略是什么？cache aside 还是 write through？

**30秒答案**:
我用的是 **Cache Aside（旁路缓存）**，这是最常用的模式。具体来说，**读的时候**先查 Redis，命中就直接返回，miss 了就查 MySQL，然后把结果写回 Redis。**写的时候**先更新 MySQL，成功后**删除** Redis 缓存，而不是更新缓存。

为什么不更新缓存？因为**并发场景下容易出脏数据**。比如两个请求同时更新，一个写缓存一个写 DB，最后缓存和 DB 对不上。删除缓存虽然会造成一次 cache miss，但下次读取会自动回填，数据一致性更好。

还有就是 **延迟双删**的问题，我在更新 MySQL 后，会先删一次缓存，等 100 毫秒再删一次，防止在这期间有旧数据被回填。这个方案在高并发场景下效果很好，数据不一致的概率降到 **0.1% 以下**。

---

### Q6: Redis 里具体存什么？key 怎么设计？

**30秒答案**:
我的 key 设计遵循一个原则：**带业务前缀和唯一标识**。比如 `user_pref:{user_id}` 存用户偏好，`session:{session_id}` 存会话数据，`vector:{query_hash}` 存向量检索结果。

这个设计有几个好处。第一是**易于管理**，可以用 `KEYS user_pref:*` 查看所有用户缓存，或者用 `DEL session:*` 批量清理会话。第二是**避免冲突**，不同业务的数据不会 key 撞车。第三是**可读性强**，看到 key 就知道是什么数据。

TTL 也根据数据特性分层设计。用户偏好变化少，TTL 设 **24 小时**。会话数据时效性强，TTL **1 小时**。向量检索结果成本高但复用性强，TTL **2 小时**。这样既能命中缓存，又不会占用太多内存。

---

### Q7: 如何避免缓存击穿 / 穿透 / 雪崩？

**30秒答案**:
这是缓存三连杀经典问题，我都做了针对性防护。

**击穿**是热点 key 过期瞬间大量请求打到 DB。我用的是**互斥锁**方案，Redis 的 SETNX 实现，只有一个请求能去查 DB，其他请求等待或者返回默认值。还有就是**逻辑过期**，不设置 TTL，而是在 value 里存过期时间，后台异步刷新。

**穿透**是查询不存在的数据。我用的是**布隆过滤器**，把所有有效的 key 存到布隆过滤器里，查询前先判断 key 是否存在。还有就是**缓存空值**，查不到的数据也缓存一个 NULL，TTL 设短一点比如 5 分钟。

**雪崩**是大量 key 同时过期。我的方案是**随机过期时间**，基础 TTL 上加 0-30% 的随机值。还有就是**缓存预热**，系统启动时把热点数据提前加载到 Redis。这三种方案组合起来，缓存稳定性提升很明显。

---

### Q8: 数据更新时如何保证缓存一致性？

**30秒答案**:
我用的是 **"先更新 DB，再删除缓存"**的策略，配合**延迟双删**。具体流程是：先更新 MySQL，成功后删除 Redis 缓存，等待 100 毫秒，再删一次缓存。

为什么这么麻烦？因为**并发场景下，只删一次会有问题**。比如请求 A 删了缓存，请求 B 查询 DB 回填了旧数据，然后请求 A 才更新 DB，这样缓存里就一直是旧数据。延迟双删能解决这个问题，第二次删除会把可能被回填的旧数据清掉。

还有就是**最终一致性**的考虑。我的系统是 AP 偏向，不需要强一致，短暂的脏数据可以接受。但通过消息队列**异步更新缓存**，保证最终一定会一致。从监控数据看，99.9% 的场景下缓存和 DB 是一致的，0.1% 的不一致也会在 1 秒内修复。

---

### Q9: 什么时候失效？TTL 怎么设计？

**30秒答案**:
我的 TTL 设计遵循**"数据变化频率 × 访问频率"**的原则。

**用户偏好**变化少但访问频繁，TTL 设 **24 小时**。这样大部分请求都能命中缓存，而且就算用户改了偏好，最多一天也会生效。实际数据显示，用户改偏好的频率平均是 **3 天一次**，所以 24 小时 TTL 是合理的。

**会话数据**时效性强，而且变化频繁，TTL **1 小时**。会话过期就清掉，不会占用内存。

**向量检索结果**成本高，相同的查询复用价值大，TTL **2 小时**。这样既能节省 LLM 调用成本，又不会让数据太旧。

还有就是**随机过期**避免雪崩，在基础 TTL 上加 0-30% 的随机值。从缓存命中率看，平均 **85%**，用户偏好能达到 **95%**，向量结果 **70%**。

---

### Q10: 如果 Redis 挂了怎么办？

**30秒答案**:
这是**高可用**的核心问题，我做了多层防护。

首先是**自动降级**。Redis 连接失败时，系统会自动切换到直连 MySQL。性能会下降，但服务不会宕机。代码里所有 Redis 操作都有 try-catch，失败就走降级路径。

其次是**健康检查**。我有一个定时任务每 5 秒 ping 一次 Redis，发现连续 3 次失败就触发告警，并且标记 Redis 为不可用。

还有**熔断保护**。如果 Redis 错误率超过 50%，就熔断 30 秒，这期间所有请求都不走 Redis，避免雪崩。

从实际运行数据看，Redis 偶尔挂掉的时候，P99 延迟从 3ms 涨到 20ms，但**服务可用性还是 99.9%**。而且 Redis 恢复后会自动回切，无需人工干预。

## 📋 母题4：性能 & 成本

**30秒答案**:
从压测数据看，**P50 延迟从 12ms 降到 0.8ms**，P99 从 50ms 降到 3ms，大概 **15 倍**的提升。

最大的提升来自缓存命中，用户偏好的命中率 **95%**，这意味着大部分请求都在 1ms 内返回。而且避免了数据库连接的开销，不用每次都建立 TCP 连接、执行查询、关闭连接。

### Q12: 命中率多少？怎么提升？

**30秒答案**:
平均 **85%**，不同数据类型差异很大。用户偏好 **95%**，因为变化少访问多。会话数据 **80%**，因为一次生成过程中会频繁访问。向量检索 **70%**，因为查询多样性高。

提升命中率的手段有几个。**缓存预热**，系统启动时把热点数据加载进去。**延长 TTL**，把用户偏好的 TTL 从 1 小时延长到 24 小时，命中率从 80% 涨到 95%。**结果复用**，相同参数的查询直接返回缓存结果。

还有就是**监控和调优**，我记录了所有 cache miss 的场景，分析原因然后针对性优化。比如发现有些 key 设计不合理，命中率低，就重新设计。

### Q13: 内存不够怎么办？

**30秒答案**:
我的策略是**只缓存热点数据**，配合 **LRU 淘汰策略**。Redis 配置了最大内存限制 2GB，满了就按 LRU 淘汰最少用的数据。

还有就是**分层缓存**，不同数据类型用不同 TTL。热数据比如用户偏好 TTL 长，冷数据比如向量结果 TTL 短。这样自然就把内存占用控制在合理范围。

从实际数据看，2GB 内存能支撑 **10 万用户**，平均每用户占用 20KB。如果用户量继续增长，可以考虑** Redis Cluster** 做水平扩展，或者**本地缓存 + Redis** 的二级缓存架构。

## 📋 母题5：LLM 专属

### Q14: 存的是"原始对话"还是"摘要后的记忆"？为什么？

**30秒答案**:
**诚实说明**：在 v5.1 架构中，我已经**移除了对话历史存储功能**，改用 LangGraph State 传递短期上下文。但我可以解释如果做会怎么设计。

如果要做，我会存**摘要后的记忆**，而不是原始对话。原因是 **token 效率**，原始对话太长，每次都塞进 prompt 成本高。**语义压缩**，提取关键信息比如用户偏好、决策理由，比原始对话更有价值。

我的方案是：实时对话用 LangGraph State 传递，Agent 间共享上下文。长期记忆只存用户偏好，比如语言、页数、语调。这样既能保持上下文连续性，又不会让 prompt 膨胀。

从数据看，只存用户偏好的话，每个用户平均 **1KB**，而存原始对话能到 **100KB**。10 万用户就是 100MB vs 10GB，差了 **100 倍**。

### Q15: 如何避免历史信息越来越多导致 prompt 爆炸？

**30秒答案**:
**诚实说明**：v5.1 中我用 **LangGraph State** 替代了对话历史存储，所以不存在这个问题。State 只保留当前任务相关的上下文，任务结束就清空。

如果需要长期记忆，我会用**滑动窗口 + 摘要**策略。保留最近 N 轮对话，更早的对话压缩成摘要。比如保留最近 5 轮原始对话，再之前的只存摘要："用户偏好专业风格，对 AI 技术感兴趣"。

还有就是**按需召回**，不是把所有历史都塞进 prompt，而是根据当前 Query 动态检索相关的历史。比如用户问"再来一个像上次那样的"，才召回上次的信息。

从实践看，滑动窗口设为 5 轮对话，足够保持上下文连续性，而 prompt 长度控制在 **2000 tokens** 以内。

### Q16: 生成结果缓存有什么用？为什么要缓存 PPT？

**30秒答案**:
主要是**节省成本**和**提升速度**。LLM 调用很贵，生成一个 PPT 的成本大概 **0.5 元**，如果用户重复生成相同内容，成本会积少成多。

我的方案是**用输入参数的 hash 做缓存 key**。比如用户两次输入完全一样的 topic 和要求，第二次就直接返回缓存结果，不需要再调用 LLM。从数据看，有 **15%** 的生成是重复或高度相似的，缓存能省不少成本。

还有就是**生成质量稳定**，相同输入一定得到相同输出，不会因为模型随机性导致差异。用户如果对某个结果满意，可以重复生成。

TTL 设 **2 小时**，因为 PPT 内容时效性一般，2 小时内重复生成的概率比较高。

## 📋 母题6：系统扩展

### Q17: 如果 10 万用户同时在线，架构还能撑住吗？

**30秒答案**:
当前的架构在 **1 万用户**规模是没问题的，但 10 万需要做一些扩展。

首先是 **Redis 扩展**，单机 Redis QPS 大概 5 万，10 万用户假设每秒 10 次操作，就是 100 万 QPS，单机扛不住。可以用 **Redis Cluster**，分 3 个主节点，每个节点 2 个副本，总 QPS 能到 **30 万+**。

其次是 **MySQL 扩展**，可以做**读写分离**，写走主库，读走从库。或者**分库分表**，按 user_id hash 到不同数据库实例。

还有就是**连接池优化**，当前 pool_size=10，可能需要调到 50，并且用连接池中间件比如 **ProxySQL**。

从成本看，支撑 10 万用户大概需要 **3 个 Redis 节点 + 1 主 2 从的 MySQL 集群**，月成本大概 **5000 元**。

### Q18: 为什么不用向量数据库做 memory？

**30秒答案**:
这是一个技术选型的 trade-off。**向量数据库**适合**语义检索**，比如"找类似的历史对话"。**MySQL** 适合**结构化数据**，比如用户偏好这种 key-value 存储。

我的场景主要是**用户个性化**，需要精确存储和读取特定偏好，比如"语言=中文，页数=15"。这种结构化数据用 MySQL 更合适，查询快、成本低、运维简单。

如果要做**语义检索**，比如"用户之前做过什么类似的主题"，那向量数据库确实更合适。可以把用户的生成记录向量化，然后做 similarity search。

但从**优先级**看，用户个性化是核心功能，语义检索是增强功能。所以我先做了 MySQL，如果未来有语义检索需求，可以再加一个向量数据库，比如 **Milvus** 或 **Weaviate**。

### Q19: 如何做多机部署？缓存如何共享？

**30秒答案**:
多机部署的核心是**共享存储层**，无状态的应用层可以水平扩展。

**Redis 共享**有几种方案。单机模式可以用**主从复制 + Sentinel**，主节点写，从节点读，Sentinel 做故障转移。大规模用 **Redis Cluster**，数据分片到多个节点，每个节点有自己的主从。

**MySQL 共享**也是类似，可以做**主从复制**，写主库，读从库。或者用 **MySQL Group Replication**，多主架构，任意节点可写。

**应用层**部署多份，用负载均衡比如 **Nginx** 做流量分发。每个应用实例都连接同一个 Redis 和 MySQL，状态是共享的。

从部署复杂度看，Nginx + 3 个应用实例 + Redis 主从 + MySQL 主从，能支撑 **5 万用户**，复杂度适中。

### Q20: 你踩过的最大缓存坑是什么？

**30秒答案**:
最大的坑是**缓存更新策略不对导致脏数据**。一开始我用的是"更新缓存"而不是"删除缓存"，结果在高并发场景下出现数据不一致。

具体场景是：用户改了偏好，我先更新 DB，然后更新缓存。但两个请求并发时，一个请求更新了 DB 但还没更新缓存，另一个请求读了旧缓存然后又写回 DB，导致 DB 的值又变成旧的。

后来改成了"删除缓存"，并且加了**延迟双删**。但一开始延迟时间设太短，50 毫秒，还是有问题。后来调到 100 毫秒，并且加了**监控告警**，缓存不一致超过 0.5% 就告警。

从这个问题学到：**缓存一致性是工程实践中的大坑**，理论上正确的方案在实际并发场景下可能还是不够，需要加监控和容错。
