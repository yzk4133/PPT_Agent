面试官脑子里的模型基本就是：

```
Request
   ↓
Redis (快)
   ↓ miss
MySQL (慢)
   ↓
回填缓存
```

然后再加：

- 失效
- 降级
- 容错
- 个性化数据结构

所以他们的问题 **80%围绕：缓存策略 + 一致性 + 稳定性**。

------

# 🔵 第一类：动机（Why 要记忆系统）

### ① 为什么 Agent 需要“记忆”？不用不行吗？

核心：业务必要性

你要能说：

- 个性化风格（模板/语气/偏好）
- 减少重复输入
- 上下文连续性
- 提升生成质量

👉 面试官在确认：不是“为了炫缓存”

------

### ② 为什么不直接每次把历史对话全塞进 prompt？

高频必问

标准痛点：

- token 爆炸
- 成本高
- latency 高
- 上下文污染
- 长对话退化

👉 这是 LLM memory 设计的经典问题

------

### ③ 为什么选“分层存储”？直接一个数据库不行吗？

考察：性能意识

你要说出：

- MySQL 延迟高（ms~10ms）
- Redis 亚毫秒
- 高频读必须走缓存

👉 体现系统优化思维

------

# 🔵 第二类：缓存设计（最高频区）

这是面试官最爱深挖的部分。

而且是区分度最大的。

------

### ④ 为什么用 Redis + MySQL 这个组合？

考察：技术选型 reasoning

期望：

- Redis：低延迟/高 QPS/内存型
- MySQL：持久化/可靠/事务支持

而不是“大家都这么用”。

------

### ⑤ 你的缓存策略是什么？cache aside 还是 write through？

高频八股 + 实战

你大概率是：
👉 Cache Aside（旁路缓存）

面试官想听到流程：

- 读：先 Redis → miss → MySQL → 回填
- 写：先 DB → 删除缓存

如果你说不出模式名称，会扣分。

------

### ⑥ Redis 里具体存什么？key 怎么设计？

非常真实的问题

比如：

```
user:{uid}:profile
user:{uid}:history
ppt:{hash}:result
```

👉 key 设计 = 工程细节体现

------

### ⑦ 如何避免缓存击穿 / 穿透 / 雪崩？

这是缓存三连杀经典题

必准备：

- 击穿 → mutex / logical expire
- 穿透 → 布隆过滤器
- 雪崩 → 随机过期时间

👉 几乎所有后端面试都会问

------

# 🔵 第三类：一致性 & 失效（进阶区）

真正做过系统的人才能答好。

------

### ⑧ 数据更新时如何保证缓存一致性？

高频

你要能讲：

- 先写 DB 再删缓存
- double delete
- eventual consistency

👉 如果只说“更新缓存”，会被认为不专业

------

### ⑨ 什么时候失效？TTL 怎么设计？

考察：策略思维

可以说：

- 用户画像：长 TTL
- 历史上下文：中 TTL
- 生成结果：短 TTL

👉 分层 TTL 非常加分

------

### ⑩ 如果 Redis 挂了怎么办？

灵魂拷问

理想答案：

- 自动降级到 MySQL
- 性能变慢但不宕机
- 熔断保护

👉 这是 production 思维

------

# 🔵 第四类：性能 & 成本（一定会问）

------

### ⑪ 延迟优化了多少？有量化数据吗？

必问

比如：

- 纯 DB：15ms
- Redis：0.5ms

👉 30x 提升

有数字就很有说服力

------

### ⑫ 命中率多少？怎么提升？

考察：指标意识

你可以说：

- 70% → 90%
- 预热
- 延长 TTL
- 结果复用

👉 “命中率”是缓存系统 KPI

------

### ⑬ 内存不够怎么办？怎么控制 Redis 占用？

高级问题

可以讲：

- LRU/LFU
- 最大内存限制
- 只缓存热点

👉 很多人答不上

------

# 🔵 第五类：LLM/Agent 专属问题（差异化亮点）

这是你这个项目特有的，会非常加分。

------

### ⑭ 你存的是“原始对话”还是“摘要后的记忆”？为什么？

考察：prompt 工程理解

高分：

- 存 summary / embedding
- 避免 token 膨胀

👉 非常 Agent 风格的问题

------

### ⑮ 如何避免历史信息越来越多导致 prompt 爆炸？

和上一题强关联

你可以讲：

- summarize
- sliding window
- top-k retrieval

👉 这是 RAG 思维

------

### ⑯ 生成结果缓存有什么用？为什么要缓存 PPT？

考察：业务优化

答案：

- 相同需求重复生成浪费算力
- hash key 直接复用

👉 体现成本意识

------

\#️# 🔵 第六类：系统设计 & 扩展（高级区）

------

### ⑰ 如果 10 万用户同时在线，架构还能撑住吗？

考察：扩展性

方向：

- Redis Cluster
- 分库分表
- sharding

------

### ⑱ 为什么不用向量数据库做 memory？

LLM 面试常见

你可以对比：

- 结构化画像 → MySQL 更合适
- 语义检索 → vector DB

👉 体现技术选型思考

------

### ⑲ 如何做多机部署？缓存如何共享？

考察：分布式理解

期望：

- 集中式 Redis
- 而不是本地内存 cache

------

### ⑳ 你踩过的最大缓存坑是什么？

真实度问题

例如：

- 脏数据
- 过期策略不合理
- key 冲突
- 内存爆掉

👉 一定准备一个“真实事故”
