## 🚀 快速开始

### 1. 了解需求
👉 先阅读 [01-需求文档.md](./01-需求文档.md)

核心要点：
- **职责**: 将用户自然语言转换为结构化需求
- **输入**: 自由文本（模板引导式）
- **输出**: PPTRequirement（Pydantic模型）
- **特性**: Few-Shot Prompt、智能推导、降级策略

### 2. 理解设计
👉 再阅读 [02-设计文档.md](./02-设计文档.md)

核心架构：
- 三层解析：LLM → JsonOutputParser → Pydantic
- 降级策略：LLM失败 → 规则提取
- 错误处理：自动重试 + 友好提示

### 3. 查看接口
👉 参考 [03-接口文档.md](./03-接口文档.md)

核心接口：
```python
agent = RequirementParserAgent()
result = await agent.parse("生成一份关于AI的PPT，15页")
```

## 🎯 核心决策总结

### 技术选型

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 解析方案 | **混合方案** | JsonOutputParser + Pydantic，平衡性能和验证 |
| Prompt策略 | **Few-Shot** | 4个示例覆盖不同场景，提升理解能力 |
| 降级策略 | **规则提取** | 保证永远返回可用结果 |
| 错误处理 | **多层防护** | 重试 + 降级 + 友好提示 |

### 架构特点

| 特性 | 说明 | 优势 |
|------|------|------|
| **三层解析** | LLM → Parser → Pydantic | 逐步验证，捕获错误 |
| **智能推导** | 场景 → 模板 → 语调 | 减少必填字段，提升体验 |
| **强制降级** | 任何失败都有保底 | 高可用性 |
| **自动重试** | 最多1次重试 | 提升成功率 |

## 💬 常见问题

**Q: 为什么选择混合方案而不是纯Pydantic？**
A: 混合方案使用简洁的格式指令，减少token消耗，同时Pydantic提供强验证，捕获LLM的"幻觉字段"。

**Q: 降级策略是否保证一定成功？**
A: 是的，规则提取使用正则和关键词匹配，保证永远返回一个可用的PPTRequirement模型。

**Q: 如何调试LLM输出？**
A: 参考使用示例中的"调试技巧"章节，可以临时替换LLM方法查看原始输出。

**Q: 支持多语言吗？**
A: 目前支持中文（ZH-CN）和英文（EN-US），通过枚举控制，便于扩展。
