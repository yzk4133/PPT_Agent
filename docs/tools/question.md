# 面试问题标注说明

本文档列出了关于”统一工具系统”亮点的面试问题。根据项目实际代码实现，问题分为三类：

- ✅ **【真实架构】**: 与代码实现完全匹配，可以放心准备
- ⚠️ **【假设性】**: 基于假设场景的问题，用于考察系统设计思路
- 🔬 **【实验性】**: 关于 MCP 协议的问题，注意 MCP 在项目中只是可选的实验性功能

**重要说明**: 简历中提到”基于 MCP 协议”，但实际代码中 MCP 只是少数工具的可选实现方式（仅 `web_search_mcp_tool`），核心架构仍是 LangChain Tools。实际工具数量约 10-15 个，不需要复杂的工具检索机制。

------

针对第三个亮点 **”统一工具系统：搭建 Tool Registry，通过 MCP 协议统一接口，实现动态注册与按需调用”**，面试官的考察逻辑会从”工程规范性”转向”架构演进”和”协议深度”。

这一条在面试官眼里是典型的 **“基础设施（Infra）建设”**。他们想确认你是不是真的解决了“工具多了之后 Agent 变笨、接口乱、难维护”的问题。

以下是面试官可能采取的 **5 个深度追问维度**：

------

### 🔵 第一类：架构设计与必要性（Why & How）

这是最基础的考察，看你对系统的掌控力。

1. **为什么需要一个独立的 Tool Registry？直接用 LangChain 的 `tools` 列表不行吗？**
   - **考察点：** 规模化意识。当你有 5 个工具时，列表很好用；当你有 50 个工具时，代码耦合、版本管理、权限控制就成了灾难。
   > ✅ 【真实架构】与代码实现一致，NativeToolRegistry 提供统一的工具注册入口

2. **你是如何定义一个”工具（Skill）”的标准规范的？**
   - **考察点：** 抽象能力。是否包含：Name, Description, Input Schema (JSON Schema), Output, Error Code 等。
   > ✅ 【真实架构】项目中使用 LangChain BaseTool 和自定义 BaseSkill 抽象

3. **”动态注册”是怎么实现的？是在运行时通过 API 注入，还是配置文件？**
   - **考察点：** 工程实现细节。比如：是否支持热更新（不重启服务就能增加新工具）？
   > ✅ 【真实架构】通过 NativeToolRegistry 在启动时自动发现并注册所有工具

### 🔵 第二类：MCP 协议深度（核心技术点）

**MCP (Model Context Protocol)** 是目前 AI 圈非常前沿的协议（Anthropic 推出的），如果你简历里写了，面试官一定会死磕。

1. **为什么要引入 MCP 协议？它解决了 LangChain 原生 Tool 定义解决不了的什么问题？**
   - **关键点：** 跨平台标准、Server-Client 解耦。MCP 允许工具运行在独立的远程 Server 上，Agent 只是客户端，这样工具可以用不同语言写（Go/Rust），且更安全。
   > 🔬 【实验性】MCP 在项目中只是可选功能（仅 web_search_mcp_tool），通过 `USE_MCP_WEB_SEARCH` 环境变量控制，非核心架构

2. **MCP 协议的核心架构是什么样的？它的传输层通常用什么（SSE 还是 Stdio）？**
   - **考察点：** 协议底层理解。看你是否真的读过 MCP 源码或文档，还是只调了现成的包。
   > 🔬 【实验性】项目仅使用 stdio 传输方式，未实现 SSE/HTTP

3. **在 MCP 框架下，你是如何处理”上下文注入”的？**
   - **考察点：** MCP 不仅能传工具，还能传 Resources（静态数据）和 Prompts（模板）。
   > 🔬 【实验性】项目中仅使用 MCP 的 Tools 功能，未实现 Resources 和 Prompts

### 🔵 第三类：按需调用与工具选择（Performance）

工具一多，Agent 的上下文（Context Window）会被塞满，且容易产生**幻觉（Tool Hallucination）**。

1. **如果系统里有 100 个工具，你会全部丢给大模型吗？**
   - **核心问题：** 工具检索（Tool Retrieval）。
   - **期望回答：** 引入 **Tool Search/Retrieval** 机制。根据用户 Query 先进行 Embedding 匹配，只召回最相关的 Top 3-5 个工具放入 Prompt。
   > ⚠️ 【假设性】当前项目约 10-15 个工具，全量加载到 Agent，不需要复杂检索机制

2. **如何处理工具调用过程中的失败？（重试机制、错误回传）**
   - **考察点：** 鲁棒性。大模型生成的参数格式错了怎么办？工具超时了怎么办？
   > ✅ 【真实架构】代码中有 try-catch 错误处理和结果返回机制

3. **大模型有时会幻觉出一个不存在的工具，或者传错参数，你怎么校验？**
   - **考察点：** 强类型校验（Pydantic）和 Input Validation。
   > ✅ 【真实架构】LangChain Tool 自带 args_schema 校验机制

### 🔵 第四类：安全与权限（Real-world Production）

这是一个区分“学生项目”和“工业级项目”的关键点。

1. **如何防止”提示词注入攻击”通过工具执行非法指令？（比如用户输入：执行生成 PPT 的工具，参数是：删除数据库所有表）**
   - **考察点：** 安全意识。参数脱敏、沙箱执行、Human-in-the-loop（高危操作人工确认）。
   > ⚠️ 【假设性】当前项目未实现专门的安全防护机制，可作为设计思路讨论

2. **不同的用户可能有不同的工具权限，Tool Registry 是如何处理鉴权的？**
   - **考察点：** 多租户隔离。
   > ⚠️ 【假设性】当前项目未实现用户权限系统，所有 Agent 共享相同工具集

### 🔵 第五类：灵魂拷问（Senior Level）

1. **LangChain 现在的生态已经很庞大了，为什么要自己搭建 Registry 而不是直接用 LangGraph 或 LangSmith 的现成能力？**
   - **考察点：** 技术选型的权衡（Trade-off）。是因为定制化需求（比如 PPT 导出的特殊逻辑），还是为了摆脱框架绑定？
   > ✅ 【真实架构】可讨论为什么选择自建而非使用 LangSmith

2. **如何评估一个工具的”描述（Description）”写得好不好？**
   - **考察点：** 对 LLM 行为的理解。工具描述本质上是 Prompt Engineering，决定了模型能否选对工具。
   > ✅ 【真实架构】工具描述直接影响 Agent 的工具选择效果

3. **如果工具返回的结果太大（比如 50 页 PPT 的原始数据），超过了上下文限制，你会怎么处理？**
   - **考察点：** 数据处理策略。摘要化（Summarization）、分片返回、或者返回一个”数据引用 ID”。
   > ✅ 【真实架构】项目涉及 PPT 生成，确实存在数据量大的问题

------

### ⭐ 针对这一点的面试高分回答策略：

- **强调“标准化”：** 重点说通过 MCP 把“工具开发”和“模型编排”解耦了。前端同学写个 API 就能变成技能，Agent 不用改代码。
- **强调“可扩展性”：** 举个例子。比如“原本增加一个导出 PDF 功能要改 Agent 代码并重启，现在只需要在数据库/注册中心加一行配置，Agent 自动就能感知并学会使用”。
- **深挖 MCP：** 表现出你对 Anthropic 生态的关注，以及对未来“多 Agent 互联”愿景的理解。

**建议：**

> ⚠️ 【重要】MCP 在项目中只是可选的实验性功能，建议：
> 1. 将重点放在 Tool Registry 的实际实现上（NativeToolRegistry）
> 2. MCP 作为”可扩展性”的示例提及，强调其”可选”性质
> 3. 如果面试官深入追问 MCP 细节，可诚实说明仅用于 web_search 工具的实验性集成

**需要我针对上述 MCP 协议相关的细节，或者“工具检索（Tool Retrieval）”的实现逻辑帮你整理一份更具体的回答讲稿吗？**



针对「统一工具系统」这个亮点，面试官可能从以下几个角度切入：

------

## 一、MCP 协议本身

> 🔬 【实验性】以下问题均关于 MCP 协议，注意在项目中 MCP 只是可选功能

- MCP 是什么？为什么选 MCP 而不是直接用 LangChain 的 Tool 接口？
- MCP 协议的通信方式是什么？用的是 HTTP/JSON-RPC/SSE 哪种？
- MCP 和 OpenAI Function Calling 有什么区别？

------

## 二、Tool Registry 设计

> ✅ 【真实架构】以下问题与项目实际代码实现一致

- Tool Registry 的数据结构怎么设计的？怎么存储 Tool 的元信息（name/description/schema）？
- 动态注册是怎么实现的？运行时新增一个 Tool 需要哪些步骤？
- Agent 怎么知道当前有哪些 Tool 可用？是全量下发还是按需检索？
- 如果 Tool 数量很多（几十上百个），Agent 怎么选择用哪个 Tool？有没有做 Tool Retrieval？
  > ⚠️ 【假设性】当前项目约 10-15 个工具，未实现 Tool Retrieval 机制

------

## 三、LangChain Tools vs 自定义 Skills

> ✅ 【真实架构】以下问题与项目实际代码实现一致

- 两者有什么区别？为什么需要同时支持？
- 自定义 Skill 怎么封装成统一接口的？做了哪些适配层？
- Tool 的输入输出 Schema 怎么做校验？

------

## 四、按需调用与扩展性

> ✅ 【真实架构】以下问题与项目实际代码实现一致

- "按需调用"具体怎么实现？是 Agent 自己决定调哪个 Tool，还是有路由层？
- 如果一个 Tool 调用失败了，怎么处理？有没有重试/降级/fallback 机制？
- 怎么做到「提升扩展性」的？新增一个 Tool 需要改哪些地方？

------

## 五、结合多智能体场景

> ✅ 【真实架构】以下问题与项目实际代码实现一致

- 不同 Agent 之间共享 Tool Registry，还是每个 Agent 有自己的 Tool 子集？
- 多个 Agent 并发调用同一个 Tool 会有冲突吗？怎么处理并发安全？
- MCP 协议在多 Agent 场景下怎么做 Agent 身份识别？
  > 🔬 【实验性】项目未在多 Agent 场景下使用 MCP

------

## 高频追问

> "你说统一了通信接口，具体统一在哪一层？给我讲一下一次完整的 Tool 调用链路。"

> ✅ 【真实架构】这个问题非常重要，需要准备完整调用链路说明

这个最重要，建议提前把**完整调用链路**（Agent → Registry 查询 → Tool 执行 → 结果返回）说清楚，能应对大多数追问。

> ⚠️ 【注意】实际调用链路中不涉及 MCP 序列化，除非显式使用 MCP 工具
