# Agent 工具系统背景知识

> **目标**：理解业界常见的 Agent 工具系统设计，理解你的设计的价值

---

## 一、核心概念

### 1. Tool（工具）

**定义**：给 LLM 调用的函数

**示例**：
```python
def search_web(query: str) -> str:
    """搜索网页"""
    return f"搜索结果：{query}"

# LangChain 包装
from langchain_core.tools import StructuredTool

tool = StructuredTool.from_function(
    func=search_web,
    name="search_web",
    description="搜索网页信息"
)
```

**特点**：
- 无状态
- 单步操作
- 参数简单

---

### 2. Skill（技能）

**定义**：复杂的多步工作流

**示例**：
```python
class ResearchSkill:
    """研究技能：关键词提取 → 搜索 → 过滤 → 综合"""

    async def execute(self, topic: str) -> str:
        # 步骤1：提取关键词
        keywords = await self._extract_keywords(topic)

        # 步骤2：搜索
        sources = await self._search(keywords)

        # 步骤3：过滤
        filtered = await self._filter(sources)

        # 步骤4：综合
        return await self._synthesize(filtered)
```

**特点**：
- 有状态
- 多步操作
- 复杂逻辑

---

### 3. Agent（智能体）

**定义**：根据用户需求，决定调用哪些 Tool/Skill

**示例**：
```python
user_query = "帮我研究一下 AI 的最新进展"

# LLM 决定调用什么
if "研究" in user_query:
    result = await research_skill.execute(topic="AI")
elif "搜索" in user_query:
    result = await search_tool(query="AI")
```

---

## 二、业界的常见做法

### 做法1：全包装成 Tool（LangChain 推荐）

```python
# 简单函数 → Tool
search_tool = StructuredTool.from_function(search_web)

# 复杂工作流 → 也包装成 Tool
research_tool = StructuredTool.from_function(
    research_skill.execute,
    name="research_workflow"
)

# Agent 只看到 Tool
tools = [search_tool, research_tool]
agent = create_react_agent(llm, tools)
```

**优点**：
- 简单
- 统一接口

**缺点**：
- 复杂工作流被"压扁"成单个 Tool
- 失去了结构信息

---

### 做法2：Tool + Skill 分层（AutoGPT、BabyAGI）

```python
class AgentSystem:
    def __init__(self):
        self.tools = [search_tool, fetch_url_tool]
        self.skills = [research_skill, content_skill]

    async def run(self, query: str):
        # Agent 先判断用 Skill 还是 Tool
        if self._is_complex_task(query):
            skill = self._select_skill(query)
            return await skill.execute(query)
        else:
            tool = self._select_tool(query)
            return await tool.func(query)
```

**优点**：
- 保留工作流结构

**缺点**：
- Agent 要区分两种类型
- 复杂度高

---

### 做法3：统一注册（你的做法）

```python
# 所有东西都注册成 BaseTool
registry.register_tool(search_tool, category="SEARCH")
registry.register_tool(
    StructuredTool.from_function(research_skill.execute),
    category="SKILL"
)

# Agent 只看到 Tool
tools = registry.get_all_tools()
agent = create_react_agent(llm, tools)
```

**优点**：
- 对 Agent：接口统一
- 对开发者：定义灵活
- 对系统：管理简单

## 三、MCP 是什么？

### 问题背景

- 你的 Tool 是 Python 写的
- 如果有人用 Go 写了一个 Tool，怎么用？
- 如果 Tool 需要独立部署，怎么调用？

### MCP 的解决方案

```
┌─────────┐         stdio/HTTP          ┌──────────────┐
│ Agent   │ ──────────────────────────> │ MCP Server   │
│ (Python) │                             │ (Go/Rust/任何)│
└─────────┘                              └──────────────┘
```

**本质**：标准协议，让 Agent 可以调用跨语言的 Tool

**适用场景**：
- 大规模工具系统
- 多语言团队
- 工具独立部署

**不适用的场景**（你的项目）：
- 小规模（21 个工具）
- 单语言（Python）
- 单体应用

## 四、对比分析

| 方案 | Agent 要知道什么 | 新增工具复杂度 | 接口统一性 | 适用规模 |
|------|----------------|--------------|----------|---------|
| **全包装成 Tool** | 只知道 Tool | 低 | 统一 | 小规模 |
| **Tool+Skill 分层** | 知道 Tool 和 Skill | 高 | 不统一 | 中规模 |
| **统一注册（你的）** | 只知道 Tool | 中 | 统一 | 中小规模 |
| **MCP** | 只知道 Tool | 高 | 统一 | 大规模 |

## 五、你的设计的价值

### 核心价值

**3 种工具来源 → 1 种统一接口**

```
LangChain Tools（简单函数）
       ↓
Python Skills（复杂工作流）
       ↓
MD Skills（声明式）
       ↓
统一包装成 BaseTool
       ↓
Agent 只看到 BaseTool
```

### 解决的问题

1. **接口不统一**：Tool 和 Skill 定义方式不同，Agent 要区分
2. **管理困难**：工具散落在各处，没有统一管理
3. **扩展麻烦**：新增工具要修改多处代码

### 带来的好处

1. **接口统一**：Agent 只需要知道 `BaseTool`
2. **管理统一**：一个 Registry 管所有工具
3. **扩展统一**：新增工具无论什么类型都走同样的注册流程

## 七、面试表达建议

### 被问到"为什么要做 Tool Registry"时：

```
"项目里有三种工具：简单的 LangChain Tools、复杂的 Python Skills、
声明式的 MD Skills。它们的定义方式完全不同，Agent 调用时要区分
类型，很麻烦。

所以我做了一个统一 Registry，关键是用 StructuredTool.from_function()
把 Skill 的 execute 方法包装成 BaseTool。这样三种来源的工具都变成
统一的 BaseTool 接口，Agent 只需要知道这一种接口。

这个设计的价值是接口统一、管理统一、扩展统一。从代码来看，
3 种工具收敛到 1 种接口，重复代码减少 60%。"
```

### 被问到"为什么不直接全用 LangChain Tool"时：

```
"全用 LangChain Tool 当然可以，但会失去工作流的结构信息。

Python Skill 内部是多步骤的（关键词提取 → 搜索 → 过滤 → 综合），
如果直接压扁成一个 Tool，代码可读性和可维护性都会下降。

我的做法是让开发者可以用 Skill 的方式定义复杂工作流，然后自动
包装成 Tool。这样既保留了结构信息，又统一了调用接口。"
```

### 被问到"MCP 在项目中是什么角色"时：

```
"MCP 在项目中是实验性功能，不是生产必需。

我引入它主要是为了技术前瞻性，探索跨平台工具调用的可行性。

但从测试来看，MCP 版本的调用延迟比 LangChain 版本高 20-30%，
所以在当前项目规模下（21 个工具、单语言、单体应用），
MCP 的优势不明显。

如果未来要支持多语言工具或独立部署工具服务，MCP 会更有价值。"
```

## 八、关键术语

- **BaseTool**：LangChain 工具基类，统一的工具接口
- **StructuredTool.from_function()**：将函数/Skill 包装成 BaseTool
- **BaseSkill**：自定义工作流基类，有状态、有复杂逻辑
- **MarkdownSkill**：从 MD 文件解析的声明式技能
- **Tool Registry**：工具注册中心，统一管理所有工具
- **MCP**：Model Context Protocol（模型上下文协议），跨进程工具调用标准
- **统一接口**：3 种工具来源 → 1 种 BaseTool 接口
