## 测试概述

### 为什么需要 Memory 层测试？

Memory 层是实现三层智能内存架构的核心组件，负责：

1. **数据存储管理** - L1/L2/L3 三层数据存储
2. **自动数据流转** - 根据访问模式和重要性自动提升数据
3. **作用域隔离** - 确保不同作用域的数据严格隔离
4. **并发安全** - 保证多线程/协程环境下的数据一致性
5. **性能优化** - LRU 缓存、批量操作、管道优化

测试的目的：

- ✅ **验证功能正确性** - 确保所有功能按预期工作
- ✅ **保证数据安全** - 防止数据泄漏、丢失或损坏
- ✅ **验证隔离性** - 确保作用域严格隔离
- ✅ **测试并发安全** - 验证线程安全机制
- ✅ **建立性能基准** - 为性能优化提供参考
- ✅ **支持重构** - 为未来代码重构提供安全网

## Memory 层架构

### 三层内存架构

```
┌─────────────────────────────────────────────────────────┐
│          HierarchicalMemoryManager                      │
│           (统一内存管理器)                                │
└──────────────┬──────────────────────────────────────────┘
               │
    ┌──────────┼──────────┬──────────────────┐
    │          │          │                  │
    ▼          ▼          ▼                  ▼
┌─────────┐ ┌─────────┐ ┌─────────┐    ┌──────────┐
│   L1    │ │   L2    │ │   L3    │    │Services  │
│瞬时内存  │ │ 短期内存│ │ 长期内存│    │          │
│         │ │         │ │         │    │ - 用户偏好│
│ - LRU   │ │ - Redis │ │ - PGSQL │    │ - 工作空间│
│ - TTL   │ │ - 管道  │ │ - 向量  │    │ - 向量搜索│
│ - 1000条│ │ - 会话级│ │ - 永久  │    │ - Agent决策│
└─────────┘ └─────────┘ └─────────┘    └──────────┘
    │          │          │
    └──────────┴──────────┴──► PromotionEngine
                                   (自动提升引擎)
```

### 内存作用域

| 作用域 | 说明 | 生命周期 | 示例 |
|--------|------|----------|------|
| **TASK** | 单个任务范围 | 秒级 | 单次 API 调用 |
| **SESSION** | 单个会话范围 | 分钟级 | 用户会话 |
| **AGENT** | 单个 Agent 范围 | 小时级 | Agent 私有数据 |
| **WORKSPACE** | 工作区范围 | 天级 | 多 Agent 协作 |
| **USER** | 用户级全局范围 | 永久 | 用户偏好设置 |

### 提升规则

#### L1 → L2 提升条件（满足其一）
- 访问次数 ≥ 3 次
- 重要性 ≥ 0.7
- 存在时间 ≥ 10 分钟 + 访问 ≥ 2 次

#### L2 → L3 提升条件（满足其一）
- 跨会话使用 ≥ 2 次
- 访问次数 ≥ 5 次 且 重要性 ≥ 0.8

## 测试策略

### Mock 策略

#### L1 层（瞬时内存）
- **策略**: 使用真实实现
- **原因**: 纯内存操作，速度快，无外部依赖
- **配置**: 小容量（100 条）用于快速测试

#### L2 层（短期内存）
- **策略**: 使用 `fakeredis`
- **原因**: 避免 Redis 服务器依赖，保证测试隔离
- **特性**: 支持所有 Redis 数据结构和命令

#### L3 层（长期内存）
- **策略**: 完全 Mock
- **原因**: 避免数据库依赖，提高测试速度
- **工具**: `unittest.mock.Mock` + `AsyncMock`

### 测试金字塔

```
        /\
       /  \        集成测试 (50+)
      /────\        - 端到端场景
     / 端到端\      - 组件交互
    /________\      - 并发安全

   /\
  /  \            单元测试 (150+)
 /────\           - 独立组件
/ 单元测试\        - 快速反馈
/__________\       - 高覆盖率
```

### 测试原则

1. **独立性** - 每个测试独立运行，无共享状态
2. **可重复性** - 多次运行结果一致
3. **快速反馈** - 单元测试秒级完成
4. **清晰命名** - 测试名称描述测试内容
5. **AAA 模式** - Arrange（准备）→ Act（执行）→ Assert（断言）

### 测试命名规范

```python
# 单元测试
async def test_{功能}_{场景}_{期望结果}()
```
