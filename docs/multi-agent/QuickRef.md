## 🔥 TOP 10 高频必考

### Q1: 为什么要用多 Agent？

**30秒答案**:
其实我一开始也试过用单 Agent，就是一个大 Prompt 一次性生成，但很快就发现问题了。首先是 **Prompt 太长**，很容易就超出了上下文限制。其次是 **一个环节出错**，整个生成过程都要重来，特别浪费时间和成本。最关键的是 **完全无法并行**，页面只能一个接一个地生成。

后来我把它拆成了 5 个 Agent，每个负责一个明确的任务，职责很清晰。这样拆分之后，我可以单独优化某个环节，比如觉得内容生成质量不好，只需要调整内容 Agent 的 Prompt 就行。更重要的是，大纲生成之后，每个页面的内容可以 **并行处理**，性能提升了 **2-3 倍**。虽然说复杂度肯定增加了，但这个收益是值得的。

---

### Q4: 你是怎么拆分 Agent 的？

**30秒答案**:
我的拆分原则其实挺简单的，就是 **按照业务流程的自然阶段来拆**。你想啊，PPT 生成这个任务，本身就有几个明显的阶段：先理解用户要什么，然后设计整体结构，如果需要的话搜集一些资料，接着生成每一页的具体内容，最后填充模板导出文件。

具体来说，我拆了 5 个 Agent。**需求解析 Agent** 负责把用户的自然语言转成结构化的需求对象，包括主题、风格、页数这些。**框架设计 Agent** 根据需求生成 PPT 的整体大纲和章节结构。**研究 Agent** 是可选的，如果用户需要数据支持或者最新资料，它会去搜集信息。**内容生成 Agent** 为每一页生成具体内容，这个是可以并行执行的。最后是 **模板渲染 Agent**，把内容填充到 PPT 模板里。

每个 Agent 都有明确的输入输出，我用 Pydantic 定义了 Schema，这样接口就很清晰，一个 Agent 的输出可以直接作为下一个 Agent 的输入，不需要额外的转换。

---

### Q5: 每个 Agent 的输入输出格式怎么设计的？

**30秒答案**:
我用 **Pydantic 定义了强类型的 Schema**，每个字段都有类型约束和验证规则。这样做最大的好处是可以自动校验，避免 LLM 输出格式错误导致整个流程崩溃。

具体来说，我定义了一套完整的数据模型。比如 Requirements 包含主题、目标受众、风格偏好、页数、是否需要研究这些字段。Outline 有章节数组和总页数。PageContent 有页码、标题、要点列表、配图建议、布局类型。

为了保证 LLM 能按照这个格式输出，我在 Prompt 里做了特殊处理。我提供了详细的 JSON Schema 作为示例，使用了 few-shot prompting 给了几个正确的样例，还加上了后处理校验，如果输出不符合 Schema，会触发重试。

用了严格的 Schema 之后，**格式错误率从 20% 降到了 5% 以下**。而且即使出错，也能快速定位到是哪个字段的定义有问题。同时有了类型定义之后，IDE 的自动补全和类型检查都能用上，开发体验好了很多。

### Q8: 为什么用 DAG 而不是简单串行？

**30秒答案**:
用 DAG 的核心原因是 **并行机会**。你想啊，大纲生成之后，每个页面的内容其实是独立的，完全可以同时生成。如果是串行的话，这些并行机会就浪费了，性能会差很多。

我用 LangGraph 的 StateGraph 构建了这个 DAG。整个流程是这样的：需求解析和框架设计是串行的，必须先执行。但内容生成阶段，大纲里的所有页面可以并行。最后模板渲染要等所有页面生成完成。除了并行，DAG 里还有一个 **条件分支**，就是研究 Agent 的调用。不是所有任务都需要搜集资料，我们通过判断需求里的一个 `need_research` 标志来决定是否走研究分支。

状态管理方面，我用了一个共享的状态对象，它在整个 DAG 流程中传递。每个节点读取自己需要的数据，更新自己负责的字段。LangGraph 会自动处理状态的传递和合并，我们不需要手动管理。

从性能数据来看，并行优化带来了显著的提升。假设一个 10 页的 PPT，每页内容生成需要 5 秒，串行的话需要 50 秒，但用 **3 个并发** 只需要大概 17 秒。整体来看，从用户输入到得到 PPT，耗时从原来的 **40 秒左右降到了 12 秒左右**，大概提升了 **2-3 倍**。

### Q10: 页面级并行具体怎么实现？

**30秒答案**:
我用的是 **异步 IO 加上信号量控制** 的并发模型。具体是用 Python 的 asyncio 库，配合 Semaphore 来限制并发数量，避免一下发出太多请求打爆 API。

在内容生成阶段，我会创建一个 Semaphore，设置最大并发数为 3。然后遍历大纲里的所有页面，为每个页面创建一个异步任务，用 asyncio.gather 并行执行。这些任务会同时启动，但由于 Semaphore 的限制，同时执行的任务最多只有 3 个，其他的会排队等待。

你可能会问，为什么用异步而不是多进程？这主要是因为异步 IO 在 **IO 密集型** 场景下效率很高，特别适合 LLM 调用这种特点。而且异步不需要复制进程空间，**内存占用小**，代码也更简单，不需要处理进程间通信。

除了 Semaphore，我还做了请求级别的限流。用 **令牌桶算法** 限制每秒钟最多发出的请求数，如果接近限流阈值，会自动加入延迟。并发度是通过环境变量配置的，可以动态调整，比如如果 API 限流比较宽松，可以调到 5 或者更高。

### Q13: 某个 Agent 失败了怎么办？

**30秒答案**:
这个问题确实很关键，我建立了一套 **多层容错机制** 来处理。首先是 **重试策略**，每个 Agent 最多重试 2 次，用的是指数退避算法，第一次重试等 2 秒，第二次等 4 秒。这样处理的是那些临时性的错误，比如网络超时、API 限流这些。

但如果重试之后还是失败，就会触发 **降级策略**，保证至少能输出一个可用的结果。具体来说，如果内容生成 Agent 失败了，就用更简化的 Prompt 重新生成，或者直接用模板填充一些占位内容。如果渲染 Agent 失败了，就先输出 Markdown 格式，至少用户能看到文字内容。如果是研究 Agent 失败了，就直接跳过研究步骤，基于需求生成内容。

还有一套 **checkpoint 机制**，每个 Agent 执行完成之后，会把中间结果持久化到数据库里，设置了 24 小时的过期时间。这样如果某个 Agent 失败了，可以从上一个 checkpoint 恢复，不用从头开始。

这套容错机制上线之后，系统的成功率从原来的 **85% 提升到了 97% 左右**。而且即使失败，用户通常也能得到一个可用的结果，而不是完全白屏。checkpoint 机制也很有用，大概 **15% 的重新生成任务** 会利用到缓存，节省了时间和成本。

### Q16: 并行带来了多大性能提升？

**30秒答案**:
从实测数据来看，并行优化带来了 **2-3 倍的整体性能提升**。具体来说，一个 10 页的 PPT，从用户输入到文件生成，耗时从原来的 **40 秒左右降到了 12 秒左右**。

我做了详细的性能分析，把整个流程分成了几个阶段来测量。串行版本的话，内容生成阶段占了总时间的 75% 以上，大概需要 50 秒，因为要一个接一个地生成每个页面。但用 3 个并发并行处理的话，只需要大概 17 秒。这个是最大的性能提升点。

除了并行优化，我还做了其他优化。**Prompt 优化** 方面，精简了指令，token 减少了 30%，API 延迟降低了 15%。**KV Cache** 方面，启用了 KV Cache 之后，推理速度提升了 2-3 倍。**连接复用** 方面，复用了 HTTP 连接，减少了握手开销。

从用户体验角度看，12 秒的等待时间是可以接受的，而 40 秒会让用户觉得慢。从成本角度看，虽然并行增加了 API 调用次数，但总时间缩短了，而且用了 KV Cache，实际成本没有增加太多。从吞吐量角度看，同样时间内能处理更多请求，服务器资源利用率更高。

### Q2: 如何判断任务"该拆"还是"不拆"？

**30秒答案**:
这个判断标准我总结为四个：**可拆分性、可并行性、可复用性、可测试性**。如果一个任务在这四个方面得分都不高，那可能就不值得拆。

**可拆分性** 是指任务能否自然地分解成独立的子任务。比如 PPT 生成，有明显的阶段划分，需求、大纲、内容、渲染，每个阶段职责清晰，适合拆分。但如果是翻译一句话这种高度耦合的任务，拆分就没有意义。

**可并行性** 是指子任务之间是否可以并行执行。大纲生成之后的页面内容，彼此独立，可以并行。如果所有子任务都必须串行执行，那多 Agent 的性能优势就体现不出来。

**可复用性** 是指拆出来的子任务是否能在其他场景复用。比如内容生成 Agent，不光能用在 PPT，还能用在文档生成、邮件撰写这些场景。如果拆出来的 Agent 只能在当前任务用，复用价值就低。

**可测试性** 是指拆分之后能否独立测试每个 Agent。如果每个 Agent 有明确的输入输出，就可以写单元测试，质量更可控。

根据这套标准，PPT 生成任务在可拆分性和可并行性上得分很高，适合拆分。而像简单的文本摘要，可能就不值得拆，因为它本身就是一个原子操作，拆分之后反而增加复杂度。

---

### Q11: 并发太多会触发限流吗？

**30秒答案**:
肯定会，这是并发系统必须考虑的问题。我用 **信号量限流加指数退避重试** 的组合来处理。信号量控制并发数，退避算法处理临时限流。

具体来说，我建了一个三层限流机制。第一层是并发控制，用 Semaphore 限制并发数为 3，这个可以通过环境变量配置。第二层是速率控制，用令牌桶算法平滑请求，避免突发。第三层是指数退避重试，如果遇到限流错误，第一次重试等 2 秒，第二次等 4 秒，第三次等 8 秒，最多重试 3 次。

除了这些预防措施，我还会记录限流错误的发生频率和时间段。如果某个时间段限流频繁，会自动降低并发度。而且并发度是支持动态调整的，不需要重启服务。

用了这些机制之后，限流导致的失败率从最初的 **15% 降到了 3% 左右**。而且即使遇到限流，大多数情况下重试一次就能成功，用户体验影响不大。并发度 3 是一个比较稳定的配置，在当前 API 限制下既能保证性能，又不会频繁触发限流。

### Q17: 哪一步最耗时？怎么定位瓶颈？

**30秒答案**:
最耗时的是 **内容生成阶段**，占了总时间的 70% 以上。我通过在每个 Agent 的入口和出口打时间戳，精确统计每个环节的耗时，来定位瓶颈。

我建了一个分层监控体系。首先是阶段级耗时统计，在每个 Agent 的入口和出口打时间戳，记录执行耗时。其次是内容生成内部的耗时细分，包括 Prompt 构造、LLM API 调用、输出解析、数据存储这些子步骤。还有一个监控面板，用 Prometheus 加 Grafana 搭建，实时展示每个 Agent 的平均耗时、P99 耗时、LLM API 的调用延迟和成功率。

通过这套监控体系，我确认内容生成是最大瓶颈，平均耗时占总时间的 75%。而且发现单页生成耗时在 3 到 8 秒之间波动，波动主要来自 LLM API 的响应时间。

针对这个发现，我们做了几个优化。**并行优化** 从 1 并发提升到 3 并发，**Prompt 优化** 让 token 减少 30%，API 延迟降低 15%，**KV Cache** 让推理速度提升 2-3 倍。最终性能提升了 **2-3 倍**。

性能优化的第一步是 **测量**，没有数据就不知道优化什么。所以我先花时间搭建完善的监控体系，虽然前期投入大，但能持续提供价值。而且性能瓶颈是动态变化的，今天的瓶颈明天可能就不是了，所以需要持续监控和调优。

## 📋 母题 1：为什么拆（架构动机）

### Q3: 多 Agent 会不会过度复杂？

**30秒答案**:
确实会，多 Agent 有 **协调成本** 和 **通信开销**。对于简单任务，单 Agent 可能更快更直接。关键是要根据任务复杂度来选择合适的架构。

我判断是否用多 Agent 主要看几个因素。一是任务是否可以拆分成独立的子任务，如果子任务之间耦合很紧，拆分反而增加复杂度。二是子任务是否可以并行执行，如果都是串行的，多 Agent 的优势就体现不出来。三是子任务是否需要不同的能力，比如搜索、计算、模板渲染，如果都需要，拆分可以各自优化。

对于 **3 页以内的简单 PPT**，我试过单 Agent 方案。就是一个 Prompt 里面包含所有指令，让模型一次生成全部内容。这种情况下，虽然可控性差一点，但速度更快，因为没有多个 Agent 之间的通信开销。

在实际系统里，我会做一个 **动态判断**。如果用户要求的页数小于 3，就走单 Agent 快速通道。如果页数大于 3 或者需要研究资料，就走多 Agent 完整流程。从测试数据看，3 页以内的 PPT，单 Agent 方案大概需要 8 秒，多 Agent 方案需要 12 秒。但 10 页的 PPT，单 Agent 可能需要 60 秒以上，而且经常出现上下文混乱的问题，多 Agent 只要 12 秒，而且质量更稳定。所以 **3 页是一个分界线**，这以内简单，这以外复杂。

架构选择的核心是 **匹配度**，不是越复杂越好。工具要匹配问题，多 Agent 是一个工具，适合复杂、可并行、可拆分的任务。简单任务用简单方案，复杂任务用复杂方案。

## 📋 母题 2：怎么拆（抽象设计）

### Q6: 多个 Agent 之间如何协同？

**30秒答案**:
我用 **共享状态对象加 Schema 约束加版本控制** 来保证协同工作。每个 Agent 读写状态的不同字段，通过 Pydantic Schema 保证数据格式一致，通过版本管理避免接口变更导致的问题。

具体来说，我定义了一个全局状态类 PPTGenerationState，包含整个流程需要共享的所有数据。有用户输入、需求对象、中间结果如大纲和研究数据、页面内容数组、最终输出文件，还有一些元数据比如当前阶段和追踪 ID。

每个 Agent 只读写自己关心的字段，避免冲突。比如需求解析 Agent 读用户输入，写需求对象。框架设计 Agent 读需求对象，写大纲。内容生成 Agent 读大纲和研究数据，写页面内容。模板渲染 Agent 读页面内容，写 PPT 文件。每个 Agent 的输入输出都很清晰，不会出现读写冲突。

关于版本管理，Schema 变更时我不会直接修改字段，而是新增字段。旧版本 Agent 继续用旧字段，新版本 Agent 用新字段，逐步迁移，等所有 Agent 都迁移完成，再删除旧字段。这样 Schema 变更时可以平滑迁移，不会影响线上服务。用了这套机制后，Agent 之间的数据传递很顺畅，几乎没有因为数据格式问题导致的失败，格式错误率控制在 **5% 以下**，版本升级时零停机。

---

### Q7: 如果只允许保留 1 个 Agent？

**30秒答案**:
我会保留 **框架设计 Agent**，然后让它在内部生成完整内容。因为框架设计是整个流程的骨架，有了结构，内容可以用更简单的方式填充。

具体实现上，我会修改框架设计 Agent 的 Prompt，让它不仅生成大纲，还直接为每个页面生成要点内容。这样一次调用就能得到完整的 PPT 结构和内容。然后直接用模板渲染 Agent 把内容填充到 PPT 里，省略了内容生成 Agent 这个环节。

如果要更简化，甚至可以把模板渲染也合并进去，让框架设计 Agent 直接生成最终的 PPT 文本格式，然后用一个简单的脚本转换成 PPT 文件。这样就真正变成了单 Agent 流程。

当然这样做的代价是可控性下降，Prompt 会变得很长，容易出错。而且无法利用并行优化，性能会受影响。但在资源受限或者快速原型验证的场景下，这种简化方案是有价值的。

从实测来看，单 Agent 版本生成 10 页 PPT 大概需要 **25 秒**，比多 Agent 的 **12 秒** 慢了一倍。但如果算上开发和维护成本，单 Agent 可能更经济。对于一些不追求极致性能的场景，单 Agent 完全够用。

这个问题的本质是 **理解最小化实现**。多 Agent 是一个优化方案，不是必须的。能拆成多 Agent，说明理解了任务的复杂性。但也能简化成单 Agent，说明理解了任务的核心逻辑。这种思维的切换很重要，避免为了技术而技术。

## 📋 母题 3：怎么调度（DAG 调度）

### Q9: DAG 不用 LangChain 能自己实现吗？

**30秒答案**:
能用，核心原理就是 **有向无环图加拓扑排序**。具体步骤是构建邻接表用字典存储节点依赖关系，计算入度统计每个节点被依赖的次数，然后用 Kahn 算法做拓扑排序，找入度为 0 的节点加入执行队列，执行该节点更新邻接节点的入度，重复直到所有节点执行完。对没有依赖关系的节点用 asyncio 并发。

如果自己实现，大概需要 **200 到 300 行代码** 就能实现基本功能。技术上没有难点，主要是工程化考虑。

我选择用 LangGraph 这个框架，主要是出于 **工程实践** 的考虑。框架已经处理了很多边界情况，比如循环依赖检测、错误传播、状态序列化这些。如果自己写，需要测试的场景会很多。而且用框架的好处是代码结构很清晰，每个节点的逻辑独立，容易测试和修改。

LangGraph 还内置了 checkpoint 机制，只需要指定一个检查点存储，就能自动保存中间状态。我们用的是内存存储，设置了 **24 小时的过期时间**。如果自己实现这些功能，需要花很多时间在各种边界情况的测试上。

当然理解框架的底层原理很重要，这样才能在出问题的时候快速定位。所以我会建议先理解原理，比如 Kahn 算法怎么实现，状态怎么传递，再使用框架。这样既能提高开发效率，又能在遇到问题的时候快速定位和解决。

## 📋 母题 4：怎么并行（并发控制）

### Q12: 100 个用户同时生成 PPT 会打爆服务吗？

**30秒答案**:
会的，当前的架构主要是单机部署，确实有并发上限。要支持 100 个并发用户，需要引入 **消息队列加 worker 池加限流降级** 的完整方案。

架构升级方面，用户请求先到 API 网关，然后进入消息队列，多个 worker 从队列取任务执行，这样可以水平扩展。具体措施包括消息队列层用 Redis Queue 或 RabbitMQ，worker 池可以部署在多台机器上实现水平扩展，worker 数量可以动态调整。

限流策略方面，用户级限流每个用户每分钟只能提交 1 个任务，全局限流系统每秒最多处理 N 个任务，超出限制返回 429 错误提示用户稍后重试。

降级策略方面，当队列长度超过阈值时启用降级模式，暂时禁用研究 Agent 减少执行时间，降低内容生成的详细程度，保证基本功能可用不会完全崩溃。还有优先级调度，付费用户的任务放高优先级队列，免费用户的任务放低优先级队列，系统空闲时才处理低优先级任务。

假设每个任务需要 **12 秒**，一个 worker 的吞吐量大概是每分钟 **5 个任务**。要支持 100 个并发用户，大概需要 **20 个 worker**。如果用 4 核机器，一台机器可以跑 **10 个 worker**，所以 **2 台机器就能撑住**。当然还要考虑峰值，实际部署可能需要 **3 到 4 台机器** 才能比较从容。

高并发设计的核心是 **分层防护**。第一层是 API 网关的限流，第二层是消息队列的缓冲，第三层是 worker 的水平扩展。每一层都能分担一部分压力，不会让单点成为瓶颈。同时降级策略保证了极端情况下的可用性，用户体验虽然会打折扣，但至少不会完全不可用。

## 📋 母题 5：怎么稳定（容错机制）

### Q14: 如何保证生成过程可恢复？

**30秒答案**:
我用 **checkpoint 机制** 来保证可恢复性。每个 Agent 执行完成后，会把中间状态持久化到数据库。如果流程中断，可以从上一个 checkpoint 恢复，不用从头开始。

Checkpoint 存储的内容包括追踪 ID、原始输入、当前执行到哪个 Agent、已完成阶段列表、完整状态对象、时间戳和 24 小时过期时间。状态对象里包含需求、大纲、已完成页面、当前做到第几页这些信息。

恢复逻辑是这样的。用户重新提交任务时，先检查是否有未完成的 checkpoint。如果有，读取 checkpoint 恢复状态，判断当前执行到哪个阶段，然后从中断的地方继续执行。如果没有 checkpoint，就从头开始。

对于长时间运行的任务，还会定期汇报进度。每完成一页，更新进度字段，前端轮询进度展示给用户，即使中断用户也能知道完成了多少。

checkpoint 机制在处理网络中断、服务重启这些场景时很有用。大概 **15% 的重新生成任务** 会利用到缓存，节省了时间和成本。而且即使服务升级重启，正在进行的任务也不会丢失，用户体验更好。

可恢复性设计的关键是 **状态持久化** 和 **幂等性**。持久化保证状态不丢失，幂等性保证重复执行不会出问题。这两个都做到，就能构建一个健壮的系统。checkpoint 的粒度也很重要，太频繁影响性能，太稀疏恢复时要做很多重复工作，需要找平衡点。我选择在 Agent 级别做 checkpoint，粒度适中，既能快速恢复，又不会太频繁。

### Q15: Agent 输出格式不稳定怎么办？

**30秒答案**:
格式不稳定是 LLM 的固有问题，我用 **强 Schema 约束加解析器加重试机制** 来应对。Pydantic 做严格校验，解析错误时提供明确反馈，最多重试 2 次。

我建了一个多层防护机制。第一层是 Prompt 层面预防，在 Prompt 里明确要求输出 JSON 格式，提供详细的格式示例，说明每个字段的类型和长度要求。第二层是 Schema 约束，用 Pydantic 定义严格的数据模型，每个字段都有类型注解和验证规则，还加了自定义验证器。第三层是智能重试，如果解析失败，把错误信息反馈给 LLM 让它重新生成，最多重试 2 次。最后一次重试失败的话，用降级策略生成兜底内容。第四层是后处理规则，截断过长的要点，补充不足的要点，过滤敏感词。

用了这套机制之后，格式错误率从 **20% 降到了 5% 以下**。即使出现错误，大多数情况下重试一次就能成功。对于顽固的错误，降级机制也能保证流程继续，不会卡死。而且通过后处理规则，即使 LLM 输出不完美，也能保证最终输出符合要求。

格式稳定性的关键是 **预期管理**。LLM 是概率模型，输出有不确定性是正常的。我们不能指望一次生成就完美，而是要建立一套纠错机制。通过 Schema 明确预期，通过解析器检测偏差，通过重试机制修正偏差，通过后处理兜底。这四层结合，才能得到稳定的输出。而且要平衡质量和效率，不是所有内容都要 100% 完美，符合用户预期就够了。

## 📋 母题 7：Agent 优化（深度调优）

### Q18: Agent 输出质量不稳定怎么优化？

**30秒答案**:
我会从 **Prompt 工程、输出约束、质量检测、自动改进** 四个层面入手。核心思路是先预防，再检测，最后补救。

Prompt 工程层面，我会加 few-shot 样例，给模型提供正确的输出示例。还会加负面示例，告诉模型不要怎么做。明确角色定位，让模型知道它扮演什么角色。加思维链引导，让模型先分析再生成。

输出约束层面，用 Pydantic 定义严格的数据模型，每个字段都有类型和长度约束。还加自定义验证器，比如检查要点不能重复，每个要点长度在 10 到 50 字之间。

质量检测层面，用另一个 LLM 调用给输出打分，评估相关性、完整性、可读性这些维度。如果评分低于 0.8，触发自动改进。

自动改进层面，把评分反馈和改进建议给 LLM，让它重新生成。最多迭代 3 次，直到质量达标。

优化之后，质量评分从 **0.75 提升到了 0.85**，格式错误率从 **20% 降到了 5% 以下**。而且即使偶尔出现质量问题，自动改进机制也能修正 **80% 的问题**，用户感知不明显。

质量优化的核心是 **多层防护**。不是指望 LLM 一次生成就完美，而是要建立一套预防加检测加改进的机制。预防可以降低问题发生率，检测可以发现问题，改进可以解决问题。这三层结合，才能保证稳定的质量输出。而且要平衡质量和成本，不是所有内容都要 100 分，符合用户预期就够了。

### Q19: 能不能用更小的模型替代大模型？

**30秒答案**:
可以，而且我正在做这件事。策略是 **分层使用模型**，非关键环节用 7B 小模型，关键质量环节用 32B 大模型，这样可以降低 **40 到 50% 的成本**。

我把 Agent 分成了三类，用不同大小的模型。简单任务用 7B 模型，包括需求解析、格式转换、质量评分。这些任务对能力要求不高，小模型完全够用。中等任务用 13B 模型，包括框架设计、模板渲染。这些任务需要一定的理解能力，但不需要最强的模型。复杂任务用 32B 模型，包括内容生成、研究资料。这些任务对质量和能力要求高，需要大模型。

为了验证小模型可行性，我做了一组对比测试。从数据看，需求解析和质量评分用 7B 模型就够了，质量损失很小但成本降低很多。框架设计用 13B 模型能在质量和成本之间取得平衡。内容生成需要保持 32B 模型，质量要求高不能降。

除了静态分层，我还设计了大小模型协作策略。先用 7B 模型快速生成草稿，再用 32B 模型做质量把关。如果质量不够，用大模型重新生成，如果质量达标就用草稿。这样可以在保证质量的前提下降低成本。

按这个策略优化后，整体成本降低了 **40 到 50%**，而质量只下降了 **3 到 5%**。模型选择的核心是 **成本收益平衡**。不是所有任务都需要最强模型，简单任务用小模型就够了。关键是要通过 A/B 测试找到每个任务的最小可行模型。而且模型技术在快速发展，现在 7B 模型的能力可能相当于半年前的 13B，所以要定期重新评估模型选择。

### Q20: 如何监控 Agent 运行状态？

**30秒答案**:
我建立了一套 **分级监控告警体系**，包括 **指标监控、日志追踪、异常检测、自动告警** 四个层次。核心是尽早发现问题，快速定位根因。

指标监控层面，我用 Prometheus 加 Grafana 搭建监控面板，采集性能指标比如 Agent 执行耗时和 LLM API 延迟，质量指标比如输出质量评分和格式错误率，成本指标比如 token 消耗量，可用性指标比如 Agent 成功率。

日志追踪层面，每个执行都有唯一的追踪 ID，记录结构化日志包括输入、输出、耗时、成功与否、token 消耗这些信息。可以按追踪 ID 查询完整执行链路，方便排查问题。

异常检测层面，我定义了多种异常情况。耗时异常指超过 P99 耗时的 2 倍，质量异常指评分低于 0.7，错误异常指连续失败 3 次，限流异常指错误率超过 10%。

自动告警层面，告警分 P0、P1、P2 三级。P0 是立即处理的级别，用 PagerDuty 和电话告警。P1 是 1 小时内处理，用邮件和 Slack。P2 是 1 天内处理，用邮件。告警消息里包含问题描述、上下文信息、处理建议。

有了这套监控告警体系，问题发现的平均时间从 **2 到 3 小时降到了 5 分钟以内**。而且 **80% 的问题** 能在用户感知之前就被发现和处理。比如某次 API 服务故障，我们在用户投诉之前就收到了告警，快速切换了备用 endpoint，用户基本无感知。

监控告警的核心是 **可观测性**。系统越复杂，越需要完善的监控体系。而且监控不是一劳永逸的，需要根据实际情况不断调整指标和阈值。告警也要避免狼来了效应，阈值要合理设置，避免误报太多导致忽略真的问题。

### Q21: 如何调试 Agent 输出？

**30秒答案**:
我建立了一套 **调试工具链**，包括 **本地调试、日志查询、输出对比、Prompt 优化** 四个层次。核心是让问题可复现、可分析、可修复。

本地调试工具可以直接运行单个 Agent，打印输入输出和耗时，还可以指定输入数据测试特定场景。日志查询系统基于 ELK Stack，可以按追踪 ID 查询完整执行链路，可视化展示每个 Agent 的执行过程。输出对比工具可以对比不同版本的输出差异和质量评分，帮助评估优化效果。Prompt 优化 SOP 是一套标准流程，包括问题分类、根因分析、优化策略、A/B 测试、灰度发布这些步骤。

有了这套调试工具链，问题定位的平均时间从 **1 到 2 小时降到了 15 到 30 分钟**。而且大部分问题可以在本地快速验证，不用每次都跑完整流程。开发效率提升很明显，原来优化一个 Agent 需要半天，现在 **1 到 2 小时** 就能搞定。

调试 Agent 的难点在于 **不确定性**。LLM 输出是概率性的，同样输入可能得到不同输出，这增加了调试难度。我的应对策略是 **增强确定性**。通过详细日志记录每次执行的上下文，让问题可复现。通过对比工具分析差异，找到优化的方向。同时要建立完善的测试样本库，覆盖各种边界情况，确保优化不会引入新问题。

## 📋 附录

### Q22: 为什么一定要生成 PPT 而不是 Markdown？

**30秒答案**:
因为 **用户需求** 和 **办公场景**。用户要的是可以直接演示的 PPT 文件，不是需要再转换的中间格式。大多数用户不熟悉 Markdown 或者 HTML，如果输出中间格式，用户还需要自己转换，增加了使用门槛。

从办公场景考虑，PPT 是商务沟通的标准工具，已经深度集成到各种工作流程里。开会投屏、发送给客户、打印讲稿，这些场景都需要 PPT 格式。

技术实现方面，虽然生成 Markdown 或 HTML 更简单，但转换成 PPT 也需要额外工作，而且转换过程可能会丢失格式、排版错乱。直接生成 PPT 质量更可控。

在内部实现里，我们确实会生成 Markdown 作为中间格式，因为这样方便调试和预览。但给用户的最终输出一定是 PPT 格式。

从实际使用来看，用户对直接输出 PPT 的反馈很好。他们不需要任何技术知识，就能拿到可用的文件。如果我们输出 Markdown，估计会有很多用户不知道怎么用，反而增加客服成本。

产品设计的核心是 **用户价值**，不是技术便利性。技术要服务于需求，而不是让用户适应技术。如果用户需要 PPT，我们就输出 PPT，即使技术上更复杂。这是做产品的第一原则。

### Q23: 商业化会优化哪三点？

**30秒答案**:
我会从 **成本优化、速度优化、个性化** 三个方向入手。成本决定能不能赚钱，速度决定用户体验，个性化决定产品竞争力。

成本优化方面，用分层模型策略。需求解析、格式转换、质量评分用 7B 模型，框架设计、模板渲染用 13B 模型，内容生成、研究资料用 32B 模型。还做 Prompt 优化，精简指令让 token 减少 30%。还有缓存策略，相似主题的页面内容可以复用，热门主题可以预生成。按估算能降低 **40 到 50%** 的 API 开支。

速度优化方面，实现流式输出，每生成一页就立即推送给用户，不等全部生成完。预生成热门主题，用户请求时直接返回缓存。用本地小模型加速，本地 7B 模型快速生成草稿，云上大模型精修。响应时间能从 **12 秒降到 3 到 5 秒**。

个性化方面，建立用户偏好系统，记录用户喜欢的风格、行业、模板、语言。根据用户偏好生成，选择匹配的模板，调整内容风格，使用用户偏好语言。还建行业模板库，教育、金融、科技等不同行业用专门的模板。个性化能提升用户留存率 **20 到 30%**。

商业化策略分免费版、个人版、企业版。免费版 3 页以内基础模板标准速度，个人版每月 9.9 元 20 页以内高级模板快速通道，企业版每月 99 元无限页数定制模板 API 接口私有部署。

商业化的核心是 **价值创造** 和 **成本控制**。用户愿意为价值付费，但价格必须合理。所以要在保证用户体验的前提下尽可能降低成本提高效率。个性化是差异化竞争的关键，让用户觉得这个产品是懂我的，这样才愿意持续付费。

### Q24: 这个项目最大的坑是什么？

**30秒答案**:
最大的坑是 **LangGraph 的状态管理**。一开始我没理解好它的状态更新机制，导致多个 Agent 之间的数据传递出了很多问题，花了很长时间才排查出来。

LangGraph 要求每个节点返回状态的 **更新**，而不是整个状态。但我一开始直接返回了新状态对象，导致其他节点写入的数据被覆盖了。这个问题很隐蔽，在某些情况下能工作，比如某些节点没写入数据的时候，在某些情况下又不行，比如多个节点都写数据的时候。而且错误信息不明确，只是说某个字段找不到了，排查起来很困难。

问题排查过程是这样的。先是发现某些时候数据会丢失，然后加了很多日志，发现是状态被覆盖。接着查阅 LangGraph 文档，理解了状态合并机制，每个节点应该只返回自己更新的字段，而不是整个状态对象。最后修复代码，问题解决。

除了这个大坑，还有一些小坑。API 的并发限制，一开始没做限流导致频繁触发限流。checkpoint 的序列化问题，某些对象不能直接序列化需要自定义转换器。异步代码的调试困难，并发执行时日志会乱需要加追踪 ID。

这个坑花了我大概 **2 到 3 天时间** 才完全解决，占项目总时间的 **15% 左右**。但解决之后，我对 LangGraph 的理解深刻了很多，后续开发就顺畅了。这也说明理解框架原理的重要性，不能只看 API 文档。

踩坑是学习过程的一部分，但要从坑里学到东西。这个坑之后，我学会了在开发新框架之前 **先看原理文档再写代码**，而不是边写边查。遇到问题 **先理解机制再改代码**，而不是盲目试错。这种思维方式的变化对我后续的开发帮助很大。

- **指数退避**：重试等待时间指数增长（2 的 n 次方）
- **令牌桶算法**：平滑请求，避免突发触发限流
- **拓扑排序**：Kahn 算法，DAG 节点执行顺序
- **Pydantic Schema**：强类型数据约束，自动校验格式

### 母题速记
1. **为什么拆**：可拆分、可并行、可复用、可测试
2. **怎么拆**：按业务流程，单一职责，Pydantic Schema
3. **怎么调度**：LangGraph StateGraph，条件分支，拓扑排序
4. **怎么并行**：asyncio 加 Semaphore 限 3，异步 IO 适合 LLM
5. **怎么稳定**：重试 2 次，降级策略，checkpoint 24 小时
6. **怎么优化**：并行 2-3 倍，KV Cache 2-3 倍，Prompt 减 30%
7. **Agent 优化**：分层模型 7B/13B/32B，质量检测，监控告警
