# 亮点一：多智能体 DAG 并行流水线

## 版本一：完整版（1-1.5 分钟）

这是我项目的核心架构。传统方式用一个大模型处理整个生成流程，但我发现这会有三个问题：一是单个 Agent 太复杂难以维护，二是无法针对特定任务优化，三是难以并行处理。

所以我用 **LangGraph** 把生成流程拆成了 **5 个专职 Agent**：需求解析、框架设计、资料研究、内容生成、模板渲染。这些 Agent 通过 **DAG 有向无环图**编排，支持两个关键特性。

第一个是**条件分支**，系统会根据任务复杂度自动判断是否需要触发资料研究。比如用户要求"生成关于人工智能的 PPT"，这种专业主题就会触发研究 Agent 去联网搜索最新资料；如果是"生成个人周报"，就直接跳过研究环节，节省时间。

第二个是**页面级并行生成**。内容生成时会创建一个异步任务池，多个页面的内容同时处理，不受前序页面阻塞。默认并发数是 3，也就是 10 页 PPT 的内容生成时间接近于 3 页的时间。

Agent 之间通过**统一状态**通信，而不是直接调用。这样避免了循环依赖，而且每个 Agent 失败时可以独立重试，不会影响其他 Agent。我还加了质量检查和改进的循环，如果内容质量不达标会自动重新生成，最多改进 3 次。

## 可能的追问

### Q1: 为什么要用多智能体，而不是一个 Agent？

**简短回答**：职责分离、易于并行、便于优化。

**详细回答**：
单一 Agent 的问题：
- 一个 Agent 要处理需求解析、框架设计、内容生成等多个任务，代码复杂度高
- 无法针对特定任务优化，比如研究 Agent 需要搜索工具，但渲染 Agent 不需要
- 无法并行处理，10 页 PPT 只能串行生成

多 Agent 的优势：
- **职责单一**：每个 Agent 只专注一件事，代码清晰，易于测试
- **并行能力**：页面级内容可以同时生成，速度提升 2-3 倍
- **灵活编排**：通过 DAG 支持条件分支，复杂任务才触发研究
- **独立优化**：每个 Agent 可以单独替换算法或调整参数

### Q2: DAG 的条件分支是怎么实现的？

**简短回答**：用 LangGraph 的 `add_conditional_edges`，根据状态字段决定下一条边。

**详细回答**：
```python
# 伪代码示例
graph.add_conditional_edges(
    "framework_designer",
    should_research,  # 判断函数
    {
        "research": "research_agent",      # 需要研究 → 研究节点
        "content": "content_agent"         # 不需要 → 直接生成
    }
)

def should_research(state):
    # 检查主题是否专业、是否需要最新数据
    if state["needs_current_data"] or state["is_professional_topic"]:
        return "research"
    return "content"
```

实际场景：
- "生成人工智能 PPT" → 需要研究（技术更新快）
- "生成个人周报" → 跳过研究（已有数据）

### Q3: 并行生成是如何实现的？会不会有冲突？

**简短回答**：用 asyncio 的 Semaphore 控制并发数，每个页面是独立任务，互不依赖。

**详细回答**：
```python
# 伪代码
class PagePipeline:
    async def generate_pages(self, state):
        pages = state["framework"]["pages"]
        semaphore = asyncio.Semaphore(3)  # 最多3个并发

        async def generate_one_page(page):
            async with semaphore:  # 限制并发数
                return await content_agent.generate(page)

        # 创建所有任务
        tasks = [generate_one_page(p) for p in pages]
        # 并行执行
        results = await asyncio.gather(*tasks)
        return results
```

**为什么不会冲突**：
- 每个页面的生成是独立的，不共享状态
- 通过 Semaphore 限制并发数，避免同时调用太多 LLM
- 如果某个页面失败，只重试该页面，不影响其他页面

### Q4: Agent 之间如何传递数据？

**简短回答**：通过 `PPTGenerationState` 这个统一状态对象，避免直接调用。

**详细回答**：
```python
class PPTGenerationState(TypedDict):
    # 输入
    user_input: str
    # 中间状态
    parsed_requirement: dict
    ppt_framework: dict
    research_data: Optional[dict]
    content_pages: list
    # 输出
    ppt_file_path: str
```

每个 Agent 接收完整的 state，处理后返回更新后的 state：
```python
class ResearchAgent:
    async def run_node(self, state: PPTGenerationState):
        # 读取需要的字段
        topic = state["ppt_framework"]["topic"]
        # 执行研究
        research_data = await self.search(topic)
        # 返回更新后的状态
        return {**state, "research_data": research_data}
```

**好处**：
- Agent 之间松耦合，不直接依赖
- 状态可以持久化，支持断点续传
- 易于调试，可以查看每个阶段的状态变化

### Q5: 如果某个 Agent 失败了怎么办？

**简短回答**：每个 Agent 有独立的错误处理，支持重试、降级、跳过三种策略。

**详细回答**：
```python
class BaseAgent:
    async def run_node(self, state):
        max_retries = 3
        for i in range(max_retries):
            try:
                result = await self.execute(state)
                return result
            except Exception as e:
                if i == max_retries - 1:
                    # 最后一次重试失败，记录错误
                    return {**state, "error": str(e)}
                logger.warning(f"Retry {i+1}/{max_retries}")
```

**三种策略**：
1. **重试**：网络超时、API 限流等临时错误
2. **降级**：高级 Agent 失败时，用简单 Agent 替代
3. **跳过**：研究 Agent 失败时，跳过研究直接生成（内容质量可能下降）

### Q6: 质量检查和改进循环是怎么做的？

**简短回答**：用 LLM 给生成内容打分，低于阈值时自动重新生成，最多 3 次。

**详细回答**：
```python
def should_refine(state):
    quality_score = state["quality_report"]["score"]
    refinement_count = state.get("refinement_count", 0)

    if quality_score < 0.8 and refinement_count < 3:
        return "refine"  # 继续改进
    return "proceed"    # 渲染 PPT
```

**评估维度**：
- 内容完整性（是否覆盖所有要点）
- 逻辑连贯性（段落之间是否流畅）
- 格式正确性（是否符合 PPT 页面要求）

**效果**：
- 85% 的内容一次通过
- 15% 需要改进 1-2 次
- 很少需要 3 次（说明提示词需要优化）

### Q7: 这个架构最大的挑战是什么？

**简短回答**：Agent 间的状态同步问题。

**详细回答**：
初期用共享内存，并发场景下数据不一致。比如并行生成多个页面时，后启动的任务可能读到旧状态。

**解决方案**：
1. **版本控制**：每个状态有 version 字段，冲突时检测
2. **不可变更新**：Agent 不修改原状态，而是返回新状态
3. **最终一致性**：允许短暂不一致，通过 MasterGraph 合并

```python
# ❌ 错误：直接修改
def agent_run(state):
    state["result"] = "done"  # 可能冲突

# ✅ 正确：返回新状态
def agent_run(state):
    return {**state, "result": "done"}
```
