# 亮点二：轻量模型训练（意图识别）

## 版本一：完整版（1-1.5 分钟）

项目初期所有意图识别都调用大模型 API，成本很高，而且延迟大约 2-3 秒。所以我训练了一个 **25.8M 参数的轻量级意图分类模型**来替代。

模型选用了 **MiniMind**，这是一个开源的超小语言模型，体积只有 GPT-3 的七千分之一。训练流程分三步：预训练、监督微调 SFT、LoRA 参数高效微调。我基于 MiniMind 的预训练权重，构建了一个包含 8 类意图的自建数据集，通过 LoRA 微调让模型学会识别用户需求。

LoRA 的核心思想是不训练全部参数，而是在每个线性层旁边加一个小的低秩矩阵。比如原本是 512×512 的权重矩阵，LoRA 只训练两个 512×8 的小矩阵，参数量从 26 万降到 4 千，减少了 98%。训练时只需要一块普通 GPU，15 分钟就能完成。

最终模型的准确率达到 **92%**，推理延迟降低到 **50 毫秒以内**，而且可以在 CPU 上运行。对于简单的意图识别场景，我会先用轻量模型处理，只有遇到复杂需求或置信度低于阈值时才调用大模型 API。这样整体成本降低了 **80%**，同时用户几乎感觉不到延迟。

## 版本二：简化版（1 分钟内）

为了降低大模型 API 调用成本，我训练了一个 **25.8M 参数的轻量意图分类模型**。

选用 **MiniMind** 作为基础模型，训练流程是：预训练 → SFT → **LoRA 微调**。LoRA 的核心是不训练全部参数，而是添加小的低秩矩阵，参数量减少 98%。

我构建了包含 8 类意图的自建数据集，用一块普通 GPU 训练 15 分钟完成。最终模型准确率 **92%**，推理延迟 **<50ms**，成本降低 **80%**。

部署时采用**混合策略**：简单场景用轻量模型，复杂场景才调用大模型，实现成本和质量的平衡。

## 可能的追问

### Q1: 为什么要自己训练模型，而不是直接用现有的分类模型？

**简短回答**：成本控制、延迟要求、数据隐私、可定制性。

**详细回答**：
1. **成本考虑**：每次调用 GPT-4 API 需要几毛钱，高频场景成本不可接受
2. **延迟要求**：API 调用需要 2-3 秒，本地推理只需要 50 毫秒
3. **数据隐私**：用户输入不应全部发送到外部 API
4. **可定制性**：现有模型难以针对特定业务场景优化

**对比表格**：

| 方案 | 成本（每次） | 延迟 | 隐私性 | 定制性 |
|-----|------------|------|--------|--------|
| GPT-4 API | ~0.1 元 | 2-3 秒 | 差 | 差 |
| 开源 BERT | 免费 | 100ms | 好 | 中 |
| **MiniMind（自训练）** | 免费 | 50ms | 好 | 好 |

### Q3: LoRA 的原理是什么？为什么能减少参数量？

**简短回答**：LoRA 在原权重旁添加低秩分解矩阵，只训练这两个小矩阵，参数量大幅减少。

**详细回答**：

**传统微调 vs LoRA**：

```
传统全量微调：
┌─────────────────────┐
│   W (512 × 512)     │  262K 参数，全部训练
│   ΔW (512 × 512)    │
└─────────────────────┘

LoRA 微调：
┌─────────────────────┐
│   W (512 × 512)     │  冻结，不训练
└─────────────────────┘
         +
┌─────────────────────┐
│   A (512 × 8)       │  4K 参数，训练 A 和 B
│   B (8 × 512)       │
└─────────────────────┘
```

**数学原理**：

权重更新可以分解为低秩矩阵：
```
ΔW = B × A
其中：W ∈ R^(d×d), A ∈ R^(r×d), B ∈ R^(d×r), r << d
```

**前向传播**：
```python
# 原始输出
h = Wx

# LoRA 输出
h = Wx + BAx = Wx + ΔWx
```

**参数量对比**：
```
全量微调：512 × 512 = 262,144 参数
LoRA (r=8): (512 × 8) + (8 × 512) = 8,192 参数
减少比例：1 - 8192/262144 ≈ 96.9%
```

**为什么有效**：
- 微调时权重变化通常集中在低秩空间
- 小的秩（r=8）足以捕捉任务特定信息
- 冻结原权重保证基础能力不变

### Q5: 训练超参数是如何选择的？

**简短回答**：基于 MiniMind 官方推荐 + 任务特点调整。

**详细回答**：

| 超参数 | 值 | 选择理由 |
|--------|-----|---------|
| **hidden_size** | 512 | MiniMind2-Small 标准配置 |
| **num_hidden_layers** | 8 | 平衡模型能力和速度 |
| **max_seq_len** | 128 | 意图识别不需要太长上下文 |
| **batch_size** | 32 | 显存允许的最大值 |
| **learning_rate** | 1e-4 | LoRA 可以用较大学习率 |
| **epochs** | 5 | 观察到 3 轮后收敛，5 轮保底 |
| **LoRA rank** | 8 | 官方推荐，平衡参数量和效果 |
| **LoRA alpha** | 16 | alpha/rank = 2 是常见选择 |
| **dropout** | 0.1 | 防止过拟合 |

**调优过程**：
1. 初始：lr=2e-4, rank=16 → 过拟合
2. 调整：lr=1e-4, rank=8 → 效果最好
3. 验证：测试集准确率 92%

### Q7: 部署时是如何混合使用的？

**简短回答**：轻量模型优先，低置信度时调用大模型。

**详细回答**：

```python
def hybrid_intent_recognition(user_input):
    # 步骤1: 轻量模型预测
    intent, confidence = minimind_model.predict(user_input)

    # 步骤2: 置信度判断
    if confidence > 0.9:
        return intent  # 高置信度，直接返回
    elif confidence > 0.7:
        # 中等置信度，用规则引擎辅助
        if rule_engine.match(user_input):
            return rule_engine.intent
        return intent
    else:
        # 低置信度，调用大模型 API
        return llm_api.classify(user_input)
```

**效果对比**：

| 方案 | 成本 | 延迟 | 准确率 |
|------|------|------|--------|
| 全部大模型 | 100% | 2-3s | 95% |
| 全部轻量模型 | 0% | 50ms | 92% |
| **混合策略** | **20%** | **100ms** | **94%** |

**策略优势**：
- 80% 请求由轻量模型处理（成本接近 0）
- 20% 复杂请求由大模型处理（保证准确率）
- 平均延迟从 2-3 秒降到 100ms

### Q8: 模型如何迭代和更新？

**简短回答**：收集错误样本 → 定期重训 → 灰度发布 → A/B 测试。

**详细回答**：

**迭代流程**：

```
1. 收集反馈

   └─ 边界 case

2. 数据增强
   ├─ 添加到训练集
   ├─ 生成变体
   └─ 验证质量

3. 增量训练
   ├─ 加载当前 LoRA 权重

4. 灰度发布
```

### Q9: 最大的挑战是什么？

**简短回答**：数据质量和边界 case 处理。

**详细回答**：

**挑战 1：数据质量**

问题：
- 初期数据样本少（每类只有 50 条）
- 表达方式单一（都是正式语气）
- 边界 case 缺失

解决方案：
- 用大模型生成变体（扩充到 500 条/类）
- 数据增强（同义词替换、句式变换）
- 人工审核边界样本

**挑战 2：相似意图混淆**

问题：
```
"生成一份 PPT" vs "查询 PPT 模板"
→ 容易混淆
```

解决方案：
- 增加对比样本
- 调整提示词格式（明确输出要求）
- 后处理规则（关键词匹配）

**挑战 3：泛化能力**

问题：
- 训练集：`"生成一份关于人工智能的 PPT"`
- 测试：`"我想做一个 AI 技术的演示"`
→ 模型可能识别错误

解决方案：
- 数据增强覆盖更多表达方式
- 增加同义词替换
- 定期用真实数据迭代
