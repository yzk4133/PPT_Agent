## 一、项目动机追问

### Q1: 为什么不直接用大模型做意图识别？

**标准答案**（30秒）：
> "主要考虑成本、延迟和 QPS。大模型 API 每次调用 $0.02，延迟 1-2 秒，还有调用频率限制。而小模型成本只有 $0.001，延迟 20ms，而且可以自己部署，没有 QPS 限制。对于意图识别这种相对简单的任务，小模型 92% 的准确率虽然略低于大模型的 97%，但换来 60 倍速度提升和 20 倍成本降低，是很划算的 trade-off。"

**答案解析**：
1. 列出具体数据（成本、延迟）
2. 对比小模型 vs 大模型
3. 强调 trade-off 是值得的
4. 体现成本意识

**可能的追问**：
- "你算过 ROI 吗？" → 见下面 Q2
- "准确率差 5% 能接受吗？" → 看业务场景，可以设计 Fallback

---

### Q2: 你怎么判断"值得自训模型"？有没有算过 ROI？

**标准答案**（30秒）：
> "算过。假设每天 10 万次调用，大模型每天成本 $2000，小模型每天 $100。训练成本一次性 $50，38 天就能回本，之后每天节省 $1900。而且小模型延迟更低，用户体验更好，还可以部署在内网保证数据安全。综合来看，对于高频调用的场景，自训模型是非常值得的。"

**答案解析**：
1. 给出具体计算（10万次 × 成本）
2. 计算回本周期（38天）
3. 提及其他收益（延迟、安全）
4. 体现商业意识

**可能的追问**：
- "如果每天只有 1 万次调用呢？" → 回本周期 380 天，可能不划算
- "训练成本 $50 怎么来的？" → GPU 租赁费用，2.25 小时 × $20/小时

**答案解析**：

### Q5: 怎么验证数据质量？会不会有过拟合问题？

**标准答案**（30秒）：
> "质量验证三层：第一层是 JSON 格式自动验证，确保字段完整；第二层是人工抽检 10%，检查合理性；第三层是训练时监控验证损失，如果异常就重新清洗。是否过拟合主要看验证损失曲线——如果训练损失持续下降但验证损失上升，就是过拟合信号。我们用的是 8:1:1 的数据划分，训练时做了早停，3 个 epoch 验证损失不改善就停止。"

**答案解析**：
1. 列出三层验证方法
2. 解释过拟合的判断标准
3. 说明数据划分比例
4. 提及早停策略

**可能的追问**：
- "验证损失上升多少算过拟合？" → 持续上升 2 个 epoch 以上
- "数据划分比例怎么定的？" → 经验值，保证每类都有样本
- "早停的 3 个 epoch 怎么定的？" → 经验值，也可以用 5

### Q10: LoRA 参数量具体怎么算？

**标准答案**（30秒）：
> "LoRA 对每个目标模块添加两个低秩矩阵 A 和 B。参数量 = rank × (input_dim + output_dim)。我们的配置 rank=8，对 q_proj、v_proj、k_proj、o_proj、gate_proj、up_proj、down_proj 都应用 LoRA。单层大约 16K 参数，8 层共 131K。相比 26M 总参数，只训练 0.5%。这是 LoRA 的核心优势：只训练极小部分参数就能达到很好的效果。"

**答案解析**：
1. 给出计算公式
2. 列出具体配置（rank=8，7个模块）
3. 给出计算结果（131K）
4. 强调占比小（0.5%）

**公式记忆**：
```
单模块参数 = rank × (hidden_size + hidden_size) = 2 × rank × hidden_size
总参数 = 单模块 × 模块数 × 层数 ≈ 2 × 8 × 512 × 7 × 8 ≈ 131K
```

**可能的追问**：
- "为什么是这 7 个模块？" → Attention 的 Q/K/V/O + FFN 的 Gate/Up/Down
- "rank=8 怎么定的？" → 经验值，是 hidden_size 的 1-2%

---

### Q11: 为什么 rank=8？不是 16 或 32？

**标准答案**（30秒）：
> "rank 是低秩矩阵的秩，越大可表达能力越强，但参数也越多。经验法则是 rank 取 hidden_size 的 1-2% 就够。对于 512 维，rank=8 是 1.5%，rank=16 是 3%。我们的实验发现 rank=8 和 rank=16 效果接近（92% vs 93%），但 rank=8 训练更快、更少过拟合。如果任务更复杂或者数据更多，可以调到 16。"

**答案解析**：
1. 解释 rank 的作用
2. 给出经验法则（1-2%）
3. 说明实验对比（rank=8 vs 16）
4. 提及调整原则

**可能的追问**：
- "有没有试过 rank=4？" → 试过，准确率 88%，不够
- "怎么判断需要调大 rank？" → 验证损失不下降或过拟合

### Q13: 训练时用的 loss 是什么？怎么计算？

**标准答案**（30秒）：
> "用的是交叉熵损失（Cross-Entropy）。因为是生成式模型，要逐 token 预测下一个 token，所以对每个位置都计算 loss。但有个关键点：我们只在 assistant 回复部分计算 loss，用户输入部分不计算。这是通过 loss_mask 实现的，mask 为 1 的位置计算 loss，为 0 的位置不计算。最终 loss 是所有有效位置的均值。"

**答案解析**：
1. 说明使用 Cross-Entropy
2. 解释逐 token 预测
3. 强调关键点：只在 assistant 部分计算
4. 提及 loss_mask 的作用

**可能的追问**：
- "为什么不用 focal loss？" → 数据均衡不需要，CE 足够
- "loss_mask 怎么实现的？" → 根据角色标记，assistant 为 1，其他为 0

## 四、性能优化追问

### Q14: 如何实现 <100ms 的推理延迟？

**标准答案**（30秒）：
> "主要三方面：第一，小模型本身快，26M 参数推理只需几毫秒；第二，KV Cache，缓存之前计算的 K/V 向量，避免重复计算，加速约 50 倍；第三，查询缓存，对常见需求直接返回，命中率 30-50%。实测平均延迟在 20-50ms，P99 < 100ms。如果还要更快，可以考虑量化或用更小的模型。"

**答案解析**：
1. 列出三个方面
2. 给出具体数据（20-50ms，命中率 30-50%）
3. 提及进一步优化方向
4. 体现工程实践能力

**可能的追问**：
- "KV Cache 怎么实现的？" → 缓存每层的 K、V 向量，推理时直接用
- "缓存怎么更新？" → LRU 策略，定期清理冷数据
- "缓存命中率怎么提升？" → 更多的缓存条目，更好的缓存策略

---

### Q15: 置信度怎么计算？阈值怎么定？

**标准答案**（30秒）：
> "置信度通过输出概率分布的 max probability 计算。生成每个 token 时取最大概率，整个序列的置信度是这些最大概率的均值。如果置信度 < 0.7，就认为不确定，触发 Fallback 到大模型。阈值 0.7 是在验证集上实验确定的，平衡准确率和 Fallback 率——太低会漏掉错误样本，太高会频繁调用大模型。实际应用中 Fallback 率在 15% 左右。"

**答案解析**：
1. 说明置信度计算方法（max probability）
2. 给出阈值（0.7）
3. 解释阈值如何确定（验证集实验）
4. 给出实际效果（Fallback 率 15%）

**可能的追问**：
- "max probability 会不会有偏差？" → 可能，所以用整个序列的均值
- "Fallback 会不会太多？" → 监控 Fallback 率，如果 > 20% 可能需要改进模型

---

### Q19: 模型误判会造成什么后果？怎么做兜底？

**标准答案**（30秒）：
> "PPT 需求分析误判可能导致生成的 PPT 不符合用户预期，影响用户体验。所以设计了三层兜底：第一层是置信度阈值，低于 0.7 调用大模型；第二层是规则引擎，对常见模式做硬编码匹配；第三层是用户反馈，误判样本会加入训练集重新训练。实际应用中误判率约 8%，通过 Fallback 机制，最终准确率能到 95%+。"

**答案解析**：
1. 说明后果（用户体验）
2. 列出三层兜底机制
3. 给出实际数据（误判率 8%，最终 95%+）
4. 体现系统可靠性思维
