## 📋 TOP 10 高频问题（优先准备）

### Q1: 为什么不直接用大模型做意图识别？

**30秒答案**:
实话实说，核心动机其实是想**学习**。当时正在系统学习大模型研发的全流程，从预训练到微调都想实践一遍。恰好 MultiAgentPPT 项目里需要识别用户想生成什么类型的 PPT，比如"教育类"还是"商业汇报"，就想**探索性地看看小模型能不能搞定这个任务**。

当然，确实也带来了成本和延迟的优化，但这更像是**完成学习目标后的附加收益**。我的思路是"先学习，再应用"，通过实际项目来验证自己掌握的技能，而不是精算 ROI 后才动手。

---

### Q2: 这个任务本质是什么问题？分类？生成？匹配？

**30秒答案**:
这个任务**不是简单的多分类**，而是**生成结构化语句**。

举个例子，用户输入"帮我做个关于AI的PPT"，我需要模型输出类似"生成一份关于人工智能的教育类PPT，风格简洁现代"这样的完整描述。具体来说，输出包含主题、类型、风格这些要素，是一个 JSON 格式的结构化输出。

所以本质上这是个 **NLG（自然语言生成）任务**，而不是给用户输入打一个"教育类"或"商业类"的标签。选择这种输出形式是因为下游系统需要**结构化的需求描述**，这样模型的输出更自然，也更容易对接后续的 PPT 生成流程。

---

### Q3: 你怎么判断"值得自训模型"？有没有算过 ROI？

**30秒答案**:
我的判断标准**不是精算 ROI，而是学习价值优先**。

首先，这个任务刚好处于"小模型可能够用"的边界——有挑战性但不是不可能。其次，我想通过实际项目来验证自己学到的模型训练知识，**看看从数据处理到 LoRA 微调的完整流程能不能跑通**。

如果做不出来，就当是一次深度学习实验；如果做成了，既能学到技术，又能帮项目降低成本。这种**探索性尝试**对我来说比算清楚"几个月回本"更有意义。当然，最终确实也证明了是值得的，否则也不会在生产环境用起来。

**如果要算 ROI 的话**，核心公式是「(月请求量 × 单次节省) - 持续成本」除以一次性投入。关键变量是月请求量——如果月 API 费用在 5000 美元以上，自训模型通常能在几个月内回本。低于这个阈值，学习价值和实验意义往往比经济回报更重要。

### Q4: 数据集类别怎么设计的？标签体系怎么定？

**30秒答案**:
我的任务是 **PPT 需求结构化提取**，不是简单分类。我设计了 **3 个维度**的类别体系：**PPT 类型 5 种**（教育培训 education、商业汇报 business、学术论文 academic、创意设计 creative、市场营销 marketing）、**风格 4 种**（极简 minimalist、现代 modern、创意 creative、专业 professional）、**目标受众 5 种**（学生 students、投资人 investors、同事 colleagues、公众 public、专家 experts）。

需要说明的是，**这三个维度在模型中不是独立预测的**。模型通过端到端生成整个 JSON，自然学习到维度间的关联（如"论文答辩"倾向生成 `academic` + `professional` + `experts`）。设计成独立字段是为了下游 Agent 的灵活性，而不是让模型做独立分类。

设计理由是分析了 MultiAgentPPT 历史请求，这套组合能覆盖 **90% 以上的真实场景**。如果太细数据难收集，太粗又不够精确，所以做了这个 trade-off。

### Q7: 为什么选 MiniMind？为什么不是 BERT / 小 LLaMA？

**30秒答案**:
选择 MiniMind 主要有三个考虑。

**学习友好性**：代码结构清晰、文档完善，社区也比较活跃，遇到问题容易找到解决方案。对于学习目的来说，这比选一个"更厉害但看不懂"的模型更合适。

**训练成本低**：参数规模只有 25.8M，在单张消费级显卡上就能完成训练，适合快速迭代实验。如果选更大的模型，光是试错成本就太高了。

**中文优化**：项目场景主要是中文输入，MiniMind 在中文语料上训练过，更适合我们的任务。综合下来，它是在学习难度、训练成本和任务适配度上的最佳平衡点。

### Q8: 预训练 + SFT + LoRA 分别在干嘛？

**30秒答案**:
这是**三阶段训练流程**，是我系统学习大模型训练的核心内容。Pretrain 用 **1.6GB 中文语料**训练，让模型学会基础的语言能力，比如理解句子结构、词语搭配。SFT 用 **1.2GB 对话数据**，让模型学会按照特定格式输出，比如 chat template。LoRA 在特定任务上微调，只训练 **rank=8 的低秩矩阵**，冻结 base model。

简单来说，pretrain 打基础，SFT 学格式，LoRA 学任务。这三个阶段我都跑通了，通过实际训练理解了每个阶段的作用和区别，而不是只看过理论。

---

### Q9: 为什么用 LoRA？直接 full fine-tune 不行吗？

**30秒答案**:
LoRA 最大的好处是**适合快速实验迭代**。Full fine-tune 需要存储所有参数的梯度，25.8M 参数也要好几百 MB 显存，而 LoRA 只训练 **rank=8 的矩阵**，显存占用减少了 **90%** 以上。训练速度也快，**6 小时**就能跑完。

对于学习目的来说，这意味着我可以**快速试错**——发现数据有问题，调整后重新训练，不需要等太久。另外 **LoRA 权重很小**，我可以为不同任务保存多个 adapter，切换时只要加载对应的权重就行了，这对探索不同任务场景非常方便。

### Q12: 延迟降低了多少？

**30秒答案**:
这个要分两层说：**核心目标达成了，附加收益也不错**。

我主要目标是学习和验证技术可行性，这一点已经做到了。至于延迟优化，确实带来了显著提升：原来调用大模型 API 需要 **1.2 秒左右**（包括网络传输），现在本地推理只需要 **20ms**，大概是 **60 倍的提升**。

但我想强调的是，**这不是我启动项目的初衷**。如果只为了优化延迟，其实有更简单的方案（比如加缓存）。我是通过学习+实践的路径，意外收获了性能优化，这更像是一举两得，而不是单一目标驱动。

### Q13: 成本降低了多少？

**30秒答案**:
成本优化同样是**附加收益，不是主要动机**。

具体数据上，原来每天大概 10 万次调用，按主流 API 定价算，每天约 **200 美元左右**。现在本地部署后，主要是计算资源成本，每天 **10 美元以内**，大概是 **20 倍的差距**。

但我想诚实地说：**我不是为了省钱才做这个项目的**。如果只算经济账，投入的精力和时间其实远大于节省的 API 费用。对我来说，核心价值是通过这个项目系统掌握了模型训练的全流程，成本优化更像是技术能力提升后的自然结果。

### Q18: 模型大小 25.8M 是怎么确定的？

**30秒答案**:
我做过**size ablation 实验**，对比了 10M、25.8M、60M 三个规模。10M 的话准确率只有 **85%**，明显不够。25.8M 达到了 **92%**，而 60M 只提升到 **94%**，但参数量翻了一倍多。

从 **性能-cost 曲线**来看，25.8M 是一个很好的拐点。再大一点的模型，性能提升有限，但显存和推理成本都显著增加。而且 25.8M 的模型可以在 **CPU 上跑**，部署灵活度更高。所以综合下来选了这个规模。

#### Q5: 数据集从哪里来？怎么构建的？

**30秒答案**:
真实用户对话只有约 **50 条**，不够训练。所以我设计了一套结构化 Prompt 模板，让 GPT-4 批量生成了 **850 条**合成数据作为初始数据集。

质量控制做了**自动化验证**（JSON 格式、字段约束）。目前数据量相对较少，但可以作为 **MVP 验证**。后续如果效果不理想，可以继续用 GPT-4 扩充到 500-1000 条。对于 LoRA 微调（rank=8）来说，几百条数据通常就足够了。

#### Q6: 数据不平衡怎么处理？

**30秒答案**:
数据量相对较小，我在生成时尽量保证**均衡分布**，每种 PPT 类型约 17 条（85 条 / 5 类），风格和受众也尽量均衡，没有严重的不平衡问题。

如果未来收集真实数据后发现不平衡，可以用 **weighted loss** 给少数类更高权重，或者用 **focal loss** 让模型更关注难样本。但目前阶段数据分布合理，暂不需要特殊处理。

#### Q6.5: 训练数据量怎么确定？多了/少了会怎样？

**30秒答案**:
数据量主要看 **3 个指标**：**验证损失收敛情况**、**训练集和验证集的损失差距**（过拟合检测）、**实际任务准确率**。

**数据太少（<50 条）**：模型会**死记硬背**，训练损失很低但验证损失很高（过拟合），泛化能力差，遇到没见过的输入就输出乱码。

**数据太多（>5000 条）**：对于 LoRA 微调（rank=8，参数量约 20K）来说，收益递减明显。训练时间变长，但准确率提升有限（可能从 92% 提到 93%）。

**经验值**：LoRA 微调通常 **200-1000 条**数据就够了。我是从 85 条开始训练，观察验证损失曲线——如果还在持续下降就继续加数据，如果已经持平说明数据量够了。

#### Q10: 训练时用的 loss 是什么？怎么计算？

**30秒答案**:
用的是 **cross-entropy loss**，但这是**生成任务**不是分类任务。模型要逐 token 生成 JSON 输出，在每个位置预测下一个 token 的概率分布，计算与真实 token 的 cross-entropy。

我用 ChatML 格式构造对话：`<|im_start|>user\n{input}<|im_end|>\n<|im_start|>assistant\n{output}<|im_end|>`。训练时只在 assistant 回复部分计算 loss（用户输入部分不计算），这样模型只学习生成结构化输出。优化器用 AdamW，learning rate 1e-4，batch size 32，训练 10 轮，损失从 2.5 降到 0.3。

#### Q11: 评估指标是什么？accuracy？F1？为什么？

**30秒答案**:
主要指标是**字段级别准确率**，因为任务是生成结构化 JSON，不是简单分类。我分别评估每个字段：topic 提取 95%、type 分类 90%、style 分类 89%、整体准确率 92%（所有字段都正确才算对）。

我还关注 **JSON 格式正确率**（能否解析出合法 JSON）和**推理性能**。验证集上整体准确率 92%，损失从 2.5 降到 0.3。对于生成任务来说，这个指标比 F1 更直观，因为用户关心的是每个字段是否正确。

### 母题五：深入扩展（4问）

#### Q15: 如果新增 10 个意图类别怎么办？要重新训练吗？

**30秒答案**:
不需要完全重新训练，可以用**增量微调**。具体来说，我会收集新类别的数据，然后用 LoRA 在原模型基础上继续训练。因为 LoRA 只训练一小块参数，**收敛很快**，大概 **2 小时**就能完成。

另外我设计了 **adapter 架构**，每个任务有独立的 LoRA 权重，新增任务只要训练一个新的 adapter 就行了，不会影响原有任务。对于真正的新类别，还可以用 **few-shot learning**，每类只要 **10-20 个样本**就能达到不错的性能。

#### Q16: 模型误判会造成什么后果？怎么做兜底？

**30秒答案**:
误判会导致 **PPT 类型识别错误**，比如把商业汇报识别成教育培训，生成的框架就不合适，需要用户修改。所以要做**兜底机制**。

我设计了一个**混合架构**：先用 MiniMind 快速解析，如果**置信度低于 0.7**，就 fallback 到 GPT-4。这样既保证大部分情况的低延迟（20ms），又对困难样本有兜底。另外还会做**人工审核**，定期抽检预测结果，监控模型性能。

#### Q17: 为什么不用 embedding + 相似度检索？

**30秒答案**:
embedding 方案不适合，因为任务是**生成结构化 JSON**，不是分类。相似度检索只能做匹配，不能生成新的内容（比如提取 topic 主题）。而且对于"极简风格"和"现代风格"这样相似的概念，embedding 很难区分。

我试过用 sentence-transformers 做相似度检索，准确率只有 **85%** 左右。而训练模型可以**端到端学习生成**，准确率达到 **92%**，多花的训练成本完全值得。另外对于字段提取任务（如从"关于AI的PPT"提取"AI"），embedding 方案根本无法实现。

## 🔑 关键数据速记

### 模型参数
- **参数规模**: 25.8M（2580 万）
- **hidden_size**: 512（大模型通常是 768-1024）
- **num_hidden_layers**: 8（大模型通常是 16-32）
- **vocab_size**: 6400（远小于标准 tokenizer 的 32K-150K）
- **num_attention_heads**: 8
- **num_key_value_heads**: 2（Grouped-Query Attention，GQA）
- **intermediate_size**: 1376（SwiGLU 激活函数）

### 性能数据
- **延迟**: 20ms（本地） vs 1200ms（API），提升 **60 倍**
- **成本**: $0.001/次（本地） vs $0.02/次（API），降低 **20 倍**
- **准确率**: 92%（小模型，字段级） vs 97%（大模型）
- **内存**: 0.5GB vs 10GB+
- **训练成本**: 约 3 RMB（$0.43）在 RTX 3090 上，6 小时

### 训练流程
1. **Pretrain**（trainer/train_pretrain.py）
   - 数据：pretrain_hq.jsonl（1.6GB 中文文本）
   - 目标：学习基础语言能力

2. **SFT**（trainer/train_full_sft.py）
   - 数据：sft_mini_512.jsonl（1.2GB 对话数据）
   - 格式：chat template（<|im_start|>user\n<|im_end|>）

3. **LoRA**（trainer/train_lora.py）
   - rank=8
   - 只训练低秩矩阵，冻结 base model
   - 支持多任务切换

### 常用术语
- **LoRA rank**: 8（低秩适应的秩）
- **DPO**: Direct Preference Optimization（直接偏好优化）
- **SwiGLU**: 激活函数变种，性能比 ReLU 好
- **GQA**: Grouped-Query Attention（分组查询注意力）
- **Cross-entropy**: 交叉熵损失，分类任务标准 loss
- **Macro-F1**: 宏平均 F1，适合不平衡数据集
- **CLS token**: 用于序列分类的特殊 token
- **Mean pooling**: 平均池化，将序列表示转换为单个向量

### 数据规模
- **Pretrain 数据**: 1.6GB 中文文本
- **SFT 数据**: 1.2GB 对话数据
- **PPT需求数据集**: 85 条（GPT-4 生成合成数据，MVP 阶段）
- **PPT 类型**: 5 种（education, business, academic, creative, marketing）
- **PPT 风格**: 4 种（minimalist, modern, creative, professional）
- **目标受众**: 5 种（students, investors, colleagues, public, experts）
- **推荐数据量**: 200-1000 条（LoRA 微调经验值）
