## 1. 基础概念

### 1.1 什么是预训练？什么是微调？

**预训练 (Pretraining)**：

```
定义: 在大规模无标注数据上训练模型，学习通用知识

数据: 维基百科、书籍、网页等
目标: 预测下一个词
损失: CrossEntropy (全部位置)
输出: 预训练权重

作用: 学习语言知识、世界知识、推理能力
```

**微调 (Fine-tuning)**：

```
定义: 在预训练基础上，用特定任务数据继续训练

数据: 任务相关数据 (如对话数据)
目标: 适配特定任务
损失: CrossEntropy (特定位置)
输入: 预训练权重

作用: 让模型适应特定任务
```

**为什么需要预训练**：

```
1. 数据效率: 从随机初始化需要更多数据
2. 知识复用: 预训练学到的知识可迁移
3. 训练稳定: 预训练提供了良好的初始化
```

**MiniMind 的两阶段训练**：

```python
# 阶段 1: 预训练
python trainer/train_pretrain.py \
    --data_path data/pretrain_hq.jsonl \
    --output out/pretrain_512.pth

# 阶段 2: SFT 微调
python trainer/train_full_sft.py \
    --data_path data/sft_data.jsonl \
    --load_from out/pretrain_512.pth \
    --output out/full_sft_512.pth
```

### 1.2 Transformer 的核心思想

**核心思想**：自注意力机制 (Self-Attention)

```
Attention(Q, K, V) = softmax(QK^T / √d_k) V

其中:
  Q (Query): 查询向量
  K (Key): 键向量
  V (Value): 值向量

含义: 计算 Q 和所有 K 的相似度，作为权重对 V 加权求和
```

**优势**：

```
1. 并行计算: 不像 RNN 需要序列计算
2. 长距离依赖: 任意两个位置都可以直接交互
3. 可解释性: 注意力权重可视化
```

**MiniMind 中的实现**：

```python
# model/model_minimind.py:167-227
scores = (xq @ xk.transpose(-2, -1)) / math.sqrt(self.head_dim)
scores = F.softmax(scores.float(), dim=-1).type_as(xq)
output = scores @ xv
```

### 1.3 为什么用 Decoder 而不是 Encoder？

**Encoder (BERT)**：

```
特点: 双向注意力，看到整个序列
适用: 分类任务 (如情感分析)
问题: 不能生成 (没有自回归)
```

**Decoder (GPT)**：

```
特点: 因果注意力，只看之前的内容
适用: 生成任务 (如对话、写作)
优势: 可以逐 token 生成
```

**MiniMind 选择 Decoder 的原因**：

```
1. 生成式模型: 可以生成对话回复
2. 通用性强: 分类、生成都能做
3. 训练简单: 因果语言建模
```

**代码体现**：

```python
# 因果掩码 (Causal Mask)
scores = scores + torch.triu(
    torch.full((seq_len, seq_len), float("-inf")),
    diagonal=1  # 只看左下角 (之前的内容)
)
```

## 2. 模型架构

### 2.1 RoPE vs 绝对位置编码

**绝对位置编码** (BERT)：

```
方法: 给每个位置分配一个固定的向量
示例: position_0 = [0.1, 0.2, ...], position_1 = [0.3, 0.4, ...]
问题: 外推能力差 (没见过训练长度外的位置)
```

**RoPE (Rotary Position Embedding)**：

```
方法: 通过旋转变换注入位置信息
核心: 注意力分数只与相对位置有关
优势: 外推能力强

代码位置: model/model_minimind.py:97-118
```

**为什么 RoPE 更好**：

```
1. 相对位置: Attention(q_m, k_n) 只依赖 (m-n)
2. 外推能力: 可以处理比训练更长的序列
3. 计算高效: 不需要额外的位置嵌入参数
```

### 2.2 GQA 的作用

**MHA (Multi-Head Attention)**：

```
Q: 8 heads
K: 8 heads
V: 8 heads
参数: 4 × d × d
```

**GQA (Grouped Query Attention)**：

```
Q: 8 heads
K: 2 heads (每 4 个 Q 头共享)
V: 2 heads
参数: 减少 ~37%
推理: 更快 (KV Cache 更小)

代码位置: model/model_minimind.py:121-130
```

**为什么使用 GQA**：

```
1. 推理加速: KV Cache 减少 75%
2. 显存节省: K, V 参数减少
3. 性能相当: 实验证明损失很小
```

### 2.3 MoE 的优势和挑战

**优势**：

```
1. 模型容量: 参数量不变但容量增加
2. 计算效率: 每个 token 只用部分专家
3. 专业化: 不同专家学习不同模式

示例: 4 个专家，每个 token 用 2 个
   参数量: 145M (和 Dense 相同)
   计算量: ~50% (只用 2/4 专家)
   容量: 4x (4 个专家)
```

**挑战**：

```
1. 训练不稳定: 专家使用不均衡
   解决: 辅助损失 (load balancing)

2. 负载均衡: 防止专家坍塌
   解决: Top-k 门控 + aux_loss

3. 通信开销: 分布式训练时专家在不同 GPU
   解决: 专家放置策略

代码位置: model/model_minimind.py:254-387
```

## 3. 训练技术

### 3.1 LoRA 原理和优势

**原理**：低秩分解

```
原权重: W ∈ R^(d×k)
LoRA: W' = W + BA, B∈R^(d×r), A∈R^(r×k), r<<min(d,k)

参数量: dr + rk ≈ 2r/d (当 k≈d)

示例: d=512, r=8
  参数比例: 2×8/512 = 3.1%
  可训练: 3.1%
  冻结: 96.9%
```

**优势**：

```
1. 参数高效: 只训练 3-10% 参数
2. 训练快速: 参数少，收敛快
3. 显存节省: 冻结大部分参数
4. 多任务: 可以为不同任务保存多个 LoRA

代码位置: model/model_lora.py
```

**vs 全参数微调**：

| 特性 | LoRA | 全参数 |
|------|------|--------|
| 参数量 | 3-10% | 100% |
| 训练时间 | 快 | 慢 |
| 显存 | 低 | 高 |
| 性能 | 略低 | 最优 |

### 3.2 DPO vs RLHF

**RLHF (Reinforcement Learning from Human Feedback)**：

```
流程:
1. 训练 Reward Model (需要大量人类标注)
2. PPO 训练 (复杂，4 个模型)
3. 需要大量计算和调参

问题: 复杂、不稳定、成本高
```

**DPO (Direct Preference Optimization)**：

```
流程:
1. 直接用偏好数据训练
2. 只需要 2 个模型 (policy + reference)
3. 更简单、更稳定

公式:
  Loss = -log sigmoid(β × (log p_θ(y) - log p_θ(x)
                            - log p_π(y) + log p_π(x)))

优势:
  - 不需要 Reward Model
  - 训练更稳定
  - 实现更简单

代码位置: trainer/train_dpo.py
```

### 3.3 混合精度训练的作用

**FP16 (16-bit Floating Point)**：

```
表示范围: 2^-14 ~ 65504
精度: 低 (有时下溢)
速度: 2x
显存: 50%
```

**BF16 (BFloat16)**：

```
表示范围: 与 FP32 相同
精度: 中等
速度: 2x
显存: 50%
推荐: 新 GPU (A100, H100)
```

**作用**：

```
1. 加速计算: GPU 对 FP16/BF16 优化更好
2. 节省显存: 权重和激活值都减半
3. 更大批次: 可以用更大的 batch size

代码位置: trainer/train_pretrain.py:176
```

**实现**：

```python
ctx = torch.cuda.amp.autocast()  # 自动混合精度
scaler = torch.cuda.amp.GradScaler()

with ctx:
    loss = model(x, y)
scaler.scale(loss).backward()
scaler.step(optimizer)
scaler.update()
```

## 4. 工程实践

### 4.1 分布式训练 (DDP)

**原理**：数据并行

```
单机: 数据 → 模型 → 输出
多机: 数据分片 → 每个GPU一份 → 各自训练 → 同步梯度

优势:
  - 加速: N 张卡 ≈ N 倍速度
  - 大批次: 可以用更大的 batch size

实现: torch.distributed (NCCL backend)

代码位置: trainer/train_pretrain.py:121-132
```

**启动命令**：

```bash
# 单机多卡
torchrun --nproc_per_node=4 trainer/train_pretrain.py --ddp

# 多机多卡
torchrun --nnodes=2 --nproc_per_node=4 trainer/train_pretrain.py --ddp
```

**注意事项**：

```
1. 只在 rank 0 保存模型和打印日志
2. 使用 DistributedSampler 而不是 shuffle=True
3. 设置不同的随机种子: seed + rank
4. 忽略 buffer 同步: _ddp_params_and_buffers_to_ignore
```

### 4.2 梯度累积的意义

**问题**：显存不够用大 batch size

```
需要: batch_size = 128
显存: 只够 batch_size = 32
```

**解决**：梯度累积

```python
accumulation_steps = 4

for step, (x, y) in enumerate(dataloader):
    loss = model(x, y) / accumulation_steps  # 缩放 loss
    loss.backward()  # 累积梯度

    if (step + 1) % accumulation_steps == 0:
        optimizer.step()  # 每 4 步更新一次
        optimizer.zero_grad()

# 有效 batch_size = 32 × 4 = 128
```

**意义**：

```
1. 显存有限: 用小 batch 模拟大 batch
2. 训练稳定: 大 batch 通常更稳定
3. 不影响性能: 梯度累积等价于大 batch
```

### 4.3 KV Cache 优化

**问题**：生成时重复计算

```
生成第 1 个 token: 计算 [token1] 的 K, V
生成第 2 个 token: 计算 [token1, token2] 的 K, V  ← token1 重复计算
生成第 3 个 token: 计算 [token1, token2, token3] 的 K, V  ← token1,2 重复计算
```

**解决**：缓存 K, V

```python
past_kv = None
for token in sequence:
    output, past_kv = model(
        token,
        past_key_value=past_kv  # 传入之前的 K, V
    )
    # past_kv 包含之前所有 token 的 K, V
```

**效果**：

```
不使用: O(N²) 计算
使用: O(N) 计算
加速: ~50x (生成长度 100)

代码位置: model/model_minimind.py:184-188
```

## 5. 对比分析

### 5.1 MiniMind vs BERT

| 特性 | MiniMind | BERT |
|------|----------|------|
| **架构** | Decoder-only | Encoder-only |
| **注意力** | 因果 (单向) | 双向 |
| **任务** | 生成式 | 分类式 |
| **训练** | 语言建模 | Masked 预测 |
| **应用** | 对话、写作 | 分类、标注 |

**选择建议**：

```
生成任务 → MiniMind (GPT)
分类任务 → BERT (如果资源充足)
         → MiniMind (小模型场景)
```

### 5.2 小模型 vs 大模型

| 特性 | 小模型 (26M) | 大模型 (7B+) |
|------|-------------|-------------|
| **训练** | 单卡消费级 GPU | 多卡 A100 |
| **推理** | CPU 可行 | 需要 GPU |
| **部署** | 边缘设备 | 服务器 |
| **能力** | 特定任务足够 | 通用能力强 |
| **成本** | 低 | 高 |

**适用场景**：

```
小模型:
  - 意图识别
  - 简单对话
  - 边缘部署
  - 快速验证

大模型:
  - 复杂推理
  - 知识密集
  - 通用助手
  - 研究探索
```

### 5.3 生成式 vs 分类式

**生成式 (MiniMind)**：

```
方法: 生成文本
输入: "识别意图：我要查询余额"
输出: "query_balance"

优势:
  - 灵活: 可以生成解释
  - 通用: 不需要重新训练
  - 可扩展: 支持多轮对话

劣势:
  - 速度: 比分类慢
  - 精度: 可能略低
```

**分类式 (传统)**：

```
方法: 分类层
输入: "我要查询余额"
输出: class_id → "query_balance"

优势:
  - 速度: 快
  - 精度: 高
  - 资源: 少

劣势:
  - 不灵活: 需要重新训练
  - 不可解释: 只给类别
```

## 6. 常见追问应答

### 6.1 为什么选择 6400 词表？

**回答框架**：

```
1. 模型规模:
   - 小模型 (26M) 用小词表
   - Embedding: 6400 × 512 = 3.3M (13% 总参数)
   - 如果 32000: 16.4M (63% 总参数) ← 太大

2. 数据规模:
   - 预训练数据相对较少
   - 小词表更不容易过拟合

3. 中文特性:
   - 中文平均每个词 1.5-2 个字符
   - 6400 覆盖大部分常用词

4. 权衡:
   - OOV (Out-of-Vocabulary): 稍多
   - 但小词表训练快、参数少
```

### 6.2 深度和宽度如何权衡？

**回答框架**：

```
1. 理论:
   - 深度: 层次特征更丰富
   - 宽度: 表达能力更强

2. 实践:
   - 太深: 训练困难 (梯度消失)
   - 太宽: 参数多、显存大

3. 经验:
   - 小模型: 中等深度 + 中等宽度
     MiniMind: 8 层 × 512 维
   - 大模型: 更深 + 更宽
     Llama-7B: 32 层 × 4096 维

4. 选择原则:
   - 任务复杂度: 复杂任务需要更深
   - 计算资源: 资源少选浅而宽
   - 实验验证: 最好的办法是实验
```

### 6.3 如何优化推理速度？

**回答框架**：

```
1. 模型层面:
   - 小模型: 用更小的模型
   - 知识蒸馏: 大模型 → 小模型
   - 剪枝: 移除不重要的权重

2. 计算层面:
   - KV Cache: 减少重复计算
   - Flash Attention: 加速注意力计算
   - 批处理: 同时处理多个请求

3. 工程层面:
   - 量化: FP16/INT8
   - 框架优化: ONNX, TensorRT
   - 缓存: 常见查询结果缓存

4. 架构层面:
   - GQA: 减少 KV Cache
   - 分层推理: 简单查询用小模型
```

### 6.4 如果训练 loss 不收敛怎么办？

**排查框架**：

```
1. 学习率:
   - 太大: loss 震荡或爆炸 → 降低学习率
   - 太小: loss 下降慢 → 提高学习率

2. 数据:
   - 质量差: 检查数据清洗
   - 格式错: 验证 JSONL 格式
   - 标签错: 检查标签一致性

3. 模型:
   - 初始化: 检查权重加载
   - 配置: 确认 vocab_size 等参数

4. 训练:
   - batch size: 太小不稳定
   - 梯度裁剪: 防止梯度爆炸
   - 混合精度: 确保 scaler 正常
```

### 6.5 如何评估生成式模型？

**评估框架**：

```
1. 自动评估:
   - 困惑度 (Perplexity): 越低越好
   - BLEU/ROUGE: 与参考文本相似度
   - 准确率: 分类任务 (如意图识别)

2. 人工评估:
   - 流畅性: 生成是否通顺
   - 相关性: 是否回答问题
   - 准确性: 信息是否正确

3. 任务评估:
   - 端到端测试: 实际应用场景
   - A/B 测试: 对比不同版本
   - 用户反馈: 真实使用数据
```

### 问题 1: 讲一下 MiniMind 的模型架构

**参考回答**：

```
MiniMind 是一个基于 Transformer Decoder 的小型语言模型。

核心组件:

特点:
- 小而精: 26M 参数，消费级 GPU 可训练

```
使用生成式方法:

### 问题 3: DPO 和 PPO 的区别？

**参考回答**：

```
PPO (传统的 RLHF):
1. 需要训练 Reward Model (需要大量标注)
2. 使用 PPO 算法 (复杂，4 个模型)
3. 训练不稳定，需要仔细调参

DPO (直接偏好优化):
1. 不需要 Reward Model
2. 直接优化偏好数据
3. 只有 2 个模型 (policy + reference)
4. 训练更稳定，实现更简单

公式:
  DPO loss = -log sigmoid(β × (policy_logratios - ref_logratios))

代码位置: trainer/train_dpo.py
```
